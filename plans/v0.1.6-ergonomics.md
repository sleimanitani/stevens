# Build Plan — v0.1.6-ergonomics: low-overhead operator surface

**Goal:** every routine operator action is **one command + at most one browser consent**. No YAML editing, no env-var juggling, no `jq`/`shred` dance. Lock the convention that the Security Agent is named **Enkidu**.

**Status:** in-progress — started 2026-04-29
**Charter refs:** STEVENS.md §3 (security architecture), §3.5 (per-install keypairs)
**Predecessor:** v0.1-sec (functionally complete, 184/184 tests green at `133dd78`)
**Successors:** Email PM live run; skills layer milestone.

---

## Scope

In this milestone:

1. Naming convention — declare **Enkidu = Security Agent**, set the convention for future agents.
2. Policy presets — `security/policy/presets/<preset>.yaml`, loader that merges into `capabilities.yaml`.
3. `stevens onboard <channel>` — single-command channel onboarding (gmail, calendar, whatsapp_cloud).
4. `stevens agent provision <name> --preset <preset>` — keypair + register + apply preset + write `.env`.
5. `stevens agent run <name>` — start the runtime sourcing the agent's `.env`.
6. `stevens passphrase remember` / `forget` — opt-in OS keyring (libsecret) integration.
7. `stevens audit tail [-f]` — JSONL tail with follow.
8. `stevens doctor` — diagnostic checklist with one-line remediations.
9. `stevens status` — running-services + last 5 audit lines.

Out of scope: skills layer, OpenClaw schema migration, WhatsApp Baileys.

### Step 2 — `stevens onboard <channel>` [x]

Replaces the multi-step `secrets add gmail.oauth_client.id` + `secrets add ...secret` + per-account `add_account` dance.

**Behavior:**
- `stevens onboard gmail --client-json ~/Downloads/client_secret_X.json --id gmail.personal --name "Sol personal"`:
  1. Reads JSON, extracts `client_id` + `client_secret`, stores both in sealed store as `gmail.oauth_client.id` / `.secret` (skips if already present unless `--rotate-client`).
  2. Shreds the source JSON file (`shred -u` semantics — multi-pass overwrite + unlink).
  3. Invokes the existing `gmail_adapter.add_account.main()` programmatically with the same args.
- `stevens onboard gmail --id gmail.work --name "Sol work"` (subsequent call): skips client setup, just runs `add_account`.
- Same shape for `calendar` (uses `calendar_adapter.add_account`) and `whatsapp_cloud` (uses `whatsapp_cloud_adapter.add_account` with `--access-token-stdin`).

**Files:**
- `security/src/stevens_security/onboard.py` — channel dispatch + shared `_shred_file()`.
- `security/src/stevens_security/cli.py` — new `onboard` subcommand.
- `security/tests/test_onboard.py` — 4 tests: client JSON ingestion (mocked sealed store), shred actually unlinks, idempotent re-onboard (`--rotate-client` required to overwrite), unknown channel → clear error.

**Reuse check:** `gmail_adapter.add_account` / `calendar_adapter.add_account` / `whatsapp_cloud_adapter.add_account` already exist. We invoke them, not reimplement.

### Step 3 — `stevens agent provision` [x]

One command for new-agent onboarding.

**Behavior:**
- `stevens agent provision email_pm --preset email_pm`:
  1. Generates Ed25519 keypair (`SigningKey.generate()` from pynacl).
  2. Writes private key to `~/.config/stevens/agents/<name>.key` (chmod 0600).
  3. Appends pubkey to `agents.yaml` (using the existing `cmd_agent_register` codepath).
  4. Merges preset into `capabilities.yaml` via `presets.merge_into_capabilities()`.
  5. Writes `~/.config/stevens/agents/<name>.env` with `STEVENS_CALLER_NAME`, `STEVENS_PRIVATE_KEY_PATH`, `STEVENS_SECURITY_SOCKET`.
  6. Prints `ready — \`stevens agent run <name>\``.
- Re-running on an existing agent: refuses unless `--force` (and `--force` requires the user to confirm a rotation, which deletes the old key and registers the new one).

**Files:**
- `security/src/stevens_security/provision.py`.
- `security/src/stevens_security/cli.py` — new `agent provision` subcommand.
- `security/tests/test_provision.py` — 5 tests: happy path writes all four artifacts, re-run without `--force` errors, unknown preset errors, env file has correct keys, key file is 0600.

**Reuse check:** keypair gen logic is in `security/scripts/gen_test_keypair.py` — extract to `provision.py`'s `_generate_keypair()`, keep the script as a thin shim that calls it for backwards compat.

### Step 4 — `stevens agent run` [x]

Wraps `python -m agents.runtime` with the agent's `.env` sourced.

**Behavior:**
- `stevens agent run email_pm`:
  1. Reads `~/.config/stevens/agents/email_pm.env` into the current process env.
  2. Verifies the key file exists and is readable (chmod 0600).
  3. Execs `python -m agents.runtime`. (Or, in dev, `uv run python -m agents.runtime` — detect via `os.environ.get("VIRTUAL_ENV")`.)

**Files:**

- `security/tests/test_agent_run.py` — 3 tests: missing .env file errors, missing key file errors, env vars set in the spawned process (mocked `os.execvpe`).

### Step 5 — `stevens passphrase remember` / `forget` [x]

Opt-in. Stores the sealed-store passphrase in the OS keyring. On every subsequent boot, Enkidu unlocks silently.

**Behavior:**
- `stevens passphrase remember` — prompts for passphrase, verifies it unlocks the store, stores in keyring under service `stevens-security`, account `vault`.
- `stevens passphrase forget` — removes from keyring.
- `stevens_security/__main__.py` consults keyring **before** falling back to `STEVENS_PASSPHRASE` env or interactive prompt.
- `_get_passphrase()` in `cli.py` also consults keyring (so `secrets add` etc. become silent too).

**Files:**
- `security/src/stevens_security/keyring_passphrase.py` — `get()`, `set(pp)`, `clear()`. Uses the `keyring` lib (Python's standard cross-platform keyring abstraction).
- `security/src/stevens_security/cli.py` — `passphrase remember/forget` subcommands.
- `security/pyproject.toml` — add `keyring>=24` dep.
- `security/tests/test_keyring_passphrase.py` — uses the in-memory `keyring.backends.fail.Keyring` fallback for tests; 3 tests: round-trip, clear, missing returns None.

**Trade-off (documented in module + STEVENS.md §3):** with keyring enabled, anyone with your unlocked desktop session can also unlock the vault. Acceptable for a single-user laptop where the threat model is "stop a compromised agent process from leaking tokens," not "stop someone with desktop access."

### Step 7 — `stevens doctor` [x]

Diagnostic checklist with one-line remediations. Returns non-zero on any failure.

**Checks:**
1. Sealed store exists at the configured root.
2. Sealed store unlocks with the available passphrase (env / keyring / prompt).
3. UDS socket exists and `ping` round-trips (only if the security service is running — otherwise note it's not running, don't fail).
4. Each agent in `agents.yaml` has a corresponding key file at the path in its `.env` (if `.env` exists in `~/.config/stevens/agents/`), and the key file is 0600.
5. Each secret name referenced in policy presets that have been merged into `capabilities.yaml` actually exists in the vault.
6. Last 24h audit log: counts of ok / deny / auth_fail per caller (informational, not pass/fail).

**Files:**
- `security/src/stevens_security/doctor.py`.
- `security/src/stevens_security/cli.py` — new `doctor` subcommand.
- `security/tests/test_doctor.py` — 5 tests covering each check returning OK and at least one returning a remediation.

## Risks

- **Keyring availability differs by host.** Linux without a Secret Service running (e.g. headless servers) can't use libsecret. The `keyring` lib has a `fail.Keyring` fallback that errors clearly — `passphrase remember` should detect and tell the user "no keyring backend available, install gnome-keyring or run with `STEVENS_PASSPHRASE` env."
- **Onboard ingestion → idempotency gotcha.** If the user re-runs `stevens onboard gmail` with a different `--client-json`, we must not silently rotate the OAuth client (it would invalidate every account that uses it). Default = error; opt in via `--rotate-client`.
- **Provision policy merge conflicts.** If the user already hand-edited `capabilities.yaml` for `email_pm` and then runs `agent provision email_pm --preset email_pm`, the merger must detect divergence and refuse to clobber.
- **Renaming churn.** Calling Enkidu by name in docs but not in code is a known-good trade-off (no migration); keep the boundary clear so future reviewers don't think the inconsistency is a bug.
