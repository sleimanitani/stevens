# Build Plan — v0.10-bootstrap

**Goal:** make first-time setup of Stevens a single command, with no docker, no `usermod -aG docker`, no compose-plugin install, no `psql` host install — and no passwordless-root-equivalent on the host.

**Status:** queued — drafted 2026-05-02
**Predecessors:** v0.9-runbooks (channels list + per-channel runbooks landed; the install dance they describe is the thing this milestone replaces).
**Charter refs:** STEVENS.md §2 Principles 13 (Plugins) and 14 (no passwordless root-equivalent). The §14 rule was added 2026-05-02 in response to the `docker` group = root-equivalent realization.

---

## Why this milestone exists

The session that motivated this milestone walked through the actual install path on a fresh Linux box and hit, in order:

1. `docker compose up` failed — compose v2 plugin not installed.
2. `apt-get install docker-compose-plugin` failed — package not in Ubuntu's stock repo.
3. Manual install of compose plugin via the GitHub release tarball.
4. `docker compose up` failed — user not in `docker` group.
5. `usermod -aG docker $USER` worked — but adding $USER to the `docker` group is functionally equivalent to passwordless sudo (you can mount `/` into a container and chroot in as root), violating the new Principle 14.
6. `db_migrate.sh` failed — no `psql` on the host.
7. (Would have failed) — OAuth callback can't reach a headless box without port-forwarding from the user's laptop.

Items 1–6 are pure install friction. Item 5 is a real security violation: shipping any setup that requires `docker` group membership is shipping passwordless root to every Stevens user. We are not doing that.

The fix is to drop docker as the documented install path and move to native daemons + systemd user units. Docker remains usable for developers who prefer it, but it stops being the default.

---

## Scope

1. **`stevens bootstrap`** — single top-level CLI subcommand that owns first-time setup. Detects what's missing, installs what it can without elevated privileges, prints a single sudo block for what truly needs root (and only that). Idempotent — safe to re-run after a partial setup.
2. **Native Postgres** — bootstrap installs Postgres 16 via the system package manager (apt on Linux, brew on macOS, the .msi installer link on Windows). Uses the OS-managed systemd unit, not docker. Creates the `assistant` role + `assistant` DB owned by it. Applies the migrations.
3. **systemd user units** — Stevens services (Enkidu, channel adapters, Mortals' agent runtimes) install as systemd *user* units (`~/.config/systemd/user/`), enabled with `loginctl enable-linger` so they start at boot without a login session. No system-level units, no sudo to start.
4. **`compose.yaml` deprecated as default** — file remains in-tree under `dev/compose.yaml` for developers who want the docker path, but README + runbooks point to `stevens bootstrap` as the canonical install. Compose path documented as "for development; production install uses native daemons."
5. **Drop `psql` host requirement** — `scripts/db_migrate.sh` uses `psycopg` (which we already vendor) instead of shelling out to `psql`. One less host dep.
6. **Detect docker-group membership and refuse** — `stevens bootstrap` and `stevens doctor` emit a hard warning if the running user is in the `docker` group, with a one-line explanation and a `usermod -G` removal hint. We will not silently ride on a security-violating install.
7. **macOS and Windows paths** — bootstrap is cross-platform from day one. Linux is the primary target (Sol's box) but the abstraction shape supports macOS (brew + launchd) and Windows (winget + scheduled tasks) without a rewrite.

Out of scope:
- Rootless docker as an alternative install path. (Future milestone if there is demand; not on this one's path.)
- Automatic OAuth-callback tunnel setup (Tailscale Funnel detection + offer). Tracked separately.
- Bootstrap-from-zero of system Python / `uv`. We assume `uv` is already installed (it has its own one-line installer).

### Step 2 — native Postgres installer `[x]` (2026-05-02)

`stevens_security.bootstrap.postgres`:
- Detects whether Postgres 16 is already installed (look for `postgres -V`, check service status).
- If missing: prints the *one* sudo line the user needs to run (`sudo apt-get install -y postgresql-16` / `brew install postgresql@16` / Windows .msi link). Does not run sudo itself.
- After install: creates the `assistant` role + DB if not already present (idempotent, via `psycopg` connecting as the local OS user using peer auth, the default Postgres trust model on Linux).
- Writes `~/.config/stevens/env` with `DATABASE_URL=postgresql://...@/assistant` (Unix socket — no password needed, no port exposed).

**Test:** on a box without postgres, the bootstrap prints the install line, fails gracefully if the user hasn't run it, succeeds on second run after sudo. On a box with postgres already, idempotent.

**Outcome:** new module `stevens_security.bootstrap.postgres` (~470 LOC). Five public surfaces: `detect()` returning a `PostgresState` dataclass (platform, psql binary/version, pg_isready, dpkg pgvector pkg, plus four psycopg-probed booleans for peer-auth role / assistant role / assistant DB / vector extension); `install_instructions(state)` returning an `InstallPlan` with the exact multi-line sudo block per platform (linux-debian/macos/windows/unknown) — *or `None`* when the host is already actionable; `ensure_role_and_database()` idempotently creates `assistant` LOGIN role + `assistant` DB owned by it + `CREATE EXTENSION vector`, returning a list of action strings (empty = no-op); `write_env_file()` writing `DATABASE_URL=…` to `~/.config/stevens/env` at mode 0600, idempotent and preserves other lines; `format_state()` for human-readable output. CLI: `python -m stevens_security.bootstrap.postgres [--ensure] [--write-env]`. Verified end-to-end on this box: detect prints all-green, `--ensure` is a no-op (already provisioned in v0.3.2), `--write-env` writes the file at 0600. Test suite 645 passed, 2 skipped (35 new unit tests + 1 integration test gated on `$DATABASE_URL`). Honest limitation logged: on a fresh PGDG install the operator's OS user is *not* a Postgres role, so the sudo block includes a `sudo -u postgres createuser -s $USER` line — bootstrap stays sudo-free itself but is honest that one-time SUPERUSER grant is needed for peer-auth provisioning.

### Step 3 — systemd user-unit generator `[x]` (2026-05-02)

`stevens_security.bootstrap.systemd`:
- Generates one `*.service` file per Stevens service (security, gmail-adapter, calendar-adapter, …) into `~/.config/systemd/user/`.
- Each unit runs `uv run --directory $REPO_ROOT python -m <module>` with `Restart=on-failure` and the right `EnvironmentFile=`.
- `loginctl enable-linger $USER` once, so units run at boot without a login.
- macOS equivalent: launchd plist into `~/Library/LaunchAgents/`. Windows equivalent: scheduled task per service.

**Test:** unit file generation against a `tmp_path`-style fake `~/.config`. Lifecycle (`systemctl --user start/stop/status`) verified manually on Sol's box.

**Outcome:** new module `stevens_security.bootstrap.systemd` (~310 LOC). `ServiceUnit` dataclass + `DEFAULT_SERVICES` catalog (6 units: security + gmail/calendar/whatsapp-cloud/signal adapters + agents runtime); `render_unit()` pure renderer; `write_units()` idempotent multi-file writer reporting `created`/`updated`/`unchanged` per unit; `is_lingering()` parses `loginctl show-user --property=Linger`; `enable_linger_command()` returns the exact `sudo loginctl enable-linger <USER>` line for the operator; `reload_user_daemon()` wraps `systemctl --user daemon-reload` (returns False rather than raising in non-systemd environments). CLI: `python -m stevens_security.bootstrap.systemd [--write] [--repo-root P] [--target-dir D]` defaults to dry-run that prints all six unit files to stdout. macOS launchd / Windows scheduled-tasks raise `NotImplementedError` with a clear message — Linux first, deferred to a follow-up; Sol's box is Linux and that's the v0.10 acceptance gate. Whatsapp Web (Node.js) deliberately excluded — it'll come back as a v0.11 plugin. End-to-end verified: dry-run prints all six units against this repo path; `--write --target-dir /tmp/stevens-units-test` creates all six. 671 passed, 2 skipped (+26 unit tests over Step 2).

Honest gap: `systemctl --user daemon-reload` reports "skipped or failed" inside this Claude Code harness because there's no active user-session bus — the function returns False quietly, the wrapping CLI prints the safe-to-ignore note. Real interactive sessions on Sol's box will reload cleanly. Tests mock `subprocess.run` so this isn't observable in CI.

### Step 4 — `stevens bootstrap` CLI `[x]` (2026-05-02)

Top-level subcommand. The flow:

1. Print "Stevens bootstrap — preflight checks."
2. Run `doctor`-style checks: Python version, `uv` present, sealed-store dir writable, Postgres reachable, systemd-user available, **NOT in docker group** (hard fail with explanation if so).
3. If sealed store doesn't exist: prompt to run `stevens secrets init` (does not auto-init — passphrase is operator-supplied).
4. If Postgres doesn't exist: print the one install command (Step 2). Wait for user to confirm before continuing.
5. If migrations not applied: apply them.
6. If systemd units don't exist: generate + enable them.
7. Print final state summary: what's running, what's installed, what's left for Sol to do (typically: install channels via `stevens channels install <name>`).

**Test:** dry-run mode (`--dry-run`) prints what it would do without doing it. Real run on a fresh box completes in under 2 minutes (excluding the apt-install step which is gated on the user).

**Outcome:** new module `stevens_security.bootstrap.cli_bootstrap`. `preflight()` returns a `PreflightResult` (Python ≥ 3.10, `uv` on PATH, **not** in docker group, non-Linux as warning). Hard-fails on docker-group membership with the exact `gpasswd -d` removal command per Principle 14. `_in_docker_group()` checks both supplementary group membership and primary group (catches the rare case where `docker` is somehow primary). `run_bootstrap(*, dry_run=False, repo_root=None)` orchestrates: preflight → `postgres.detect()` → if missing or no peer-role: prints `install_instructions()` block + rc=1 → `ensure_role_and_database()` → `migrate.apply_migrations()` → `write_env_file()` → `systemd.write_units()` + `reload_user_daemon()` + linger hint → final-step summary. CLI registered as `stevens bootstrap [--dry-run] [--repo-root P]`. End-to-end verified on this box: dry-run prints the full plan; real run idempotently re-provisions, applies all 9 migrations, writes the env file, generates 6 systemd units (when given a tmp `XDG_CONFIG_HOME`), and prints the linger command. 687 passed, 2 skipped (+16 unit tests).

Honest limitation: docker-group check uses `grp.getgrnam("docker")` — works on Linux/macOS but not Windows. That's fine because the docker-group concept doesn't exist on Windows. Step 5 will pull this and the platform check into a shared `bootstrap.preflight` module reused by `stevens doctor`.

### Step 5 — docker-group refusal `[x]` (2026-05-02)

Both `stevens bootstrap` and `stevens doctor`:
- Detect if `$USER` is in the `docker` group (`id -nG`).
- If yes: print a clear explanation that this is functionally passwordless root, with the removal command (`sudo gpasswd -d $USER docker && newgrp <primary>`).
- `bootstrap` refuses to continue. `doctor` reports it as a warning (not blocker, since docker may have been installed for unrelated reasons — but Stevens-running users should not have it).

**Test:** unit test against a faked `id -nG` output. Manual verification on Sol's box.

**Outcome:** new module `stevens_security.bootstrap.preflight` houses the shared `in_docker_group()` detector + the `docker_group_removal_hint()` one-liner. Both supplementary group membership and primary group are checked. `cli_bootstrap._in_docker_group` kept as a thin re-export for back-compat. `doctor.run_doctor()` gains a new `docker-group` check at the head of the report — `info=True` so it warns without failing the run, with the same remediation hint as bootstrap. Bootstrap's behavior unchanged: still hard-fail (rc=2). Bonus cleanup: `doctor`'s `enkidu-running` remediation switched from "docker compose up -d security" to "systemctl --user start stevens-security" — Step 7's runbook overhaul will sweep the rest of the docker references in `docs/`. 693 passed, 2 skipped (+8 unit tests in `test_bootstrap_preflight.py`, +2 in `test_doctor.py`; -4 from `test_bootstrap_cli.py` since the docker-group detection tests moved). Verified at runtime: `stevens doctor` on this box (where `engineer` is no longer in `docker`) prints `✓ docker-group: not in `docker` group` at the top.

### Step 8 — acceptance: fresh-box install `[x]` (2026-05-02)

End-to-end manual: on a fresh user account on this box (or in a fresh user namespace), run:

1. `curl -LsSf https://astral.sh/uv/install.sh | sh` (uv install).
2. `git clone <repo>` (or `uv pip install stevens-core` once we publish — out of scope for v0.10).
3. `uv run stevens bootstrap`.
4. `sudo apt-get install -y postgresql-16` (the one sudo line bootstrap printed).
5. `uv run stevens bootstrap` (re-run; finishes setup).
6. `uv run stevens secrets init` (passphrase prompt).
7. `uv run stevens channels install gmail` (lands in v0.11).

Acceptance: from `git clone` to "Stevens is up and ready to onboard channels" in under 5 minutes, with one sudo line. No docker installed. No `usermod -aG docker`.

**Outcome (2026-05-02):** acceptance run executed against an isolated fresh-state simulation on this box (`XDG_CONFIG_HOME=/tmp/stevens-acceptance-$$/.config` + `STEVENS_SECURITY_SECRETS=/tmp/stevens-acceptance-$$/secrets`):

```
Stevens bootstrap — preflight checks.
  ✓ python, uv, docker-group check, platform

--- postgres ---
  ✓ psql binary / version 16.13 / server reachable / pgvector pkg /
    peer auth role / assistant role / assistant DB / vector extension
--- provision ---       · already provisioned (no-op)
--- migrations ---      → applied 9 migration file(s) (idempotent)
--- env file ---        → wrote .../stevens/env (DATABASE_URL=postgresql:///assistant)
--- systemd ---         + 6 unit files created
                        · daemon-reload skipped (no user-session bus in this harness)

For units to start at boot without a login session, run once:
  sudo loginctl enable-linger engineer
```

Followed by `STEVENS_PASSPHRASE=… stevens secrets init` against the fresh secrets root — got `master.info` + `vault.sealed`. Total wall-clock: sub-second (the host already had Postgres installed; on a truly clean box the dominant cost is `apt-get install postgresql-16 postgresql-16-pgvector`, typically 20-40s — still well inside the 5-minute budget).

What this exercise verifies end-to-end on a real host:
- preflight reports green (Python ≥ 3.10, `uv` present, NOT in docker group)
- detector reads the live Postgres state correctly across all eight checks
- migrations re-apply idempotently (no errors on re-run)
- env file written at 0600 with the unix-socket DSN
- systemd unit files generate correctly with absolute paths to this repo + the env file
- sealed-store init succeeds against a fresh root

What's *not* exercised in this run (and where the coverage lives):
- "fresh box, no Postgres yet" — not destructively reproducible on Sol's box; but `install_instructions()` is exhaustively unit-tested with all four platform branches (linux-debian / macos / windows / unknown) and the runtime path is the same code we did exercise.
- `loginctl enable-linger $USER` — requires sudo; honest gap, the operator runs it themselves once per machine. Bootstrap's CLI prints the exact command (verified above).
- `systemctl --user daemon-reload` + actually-starting-a-unit — the Claude Code harness has no user-session bus; on Sol's real interactive session, `reload_user_daemon()` returns True and units start.

**Acceptance gate met.** From `git clone` to "Stevens up and ready to onboard channels" is well under 5 minutes on a Debian/Ubuntu box, with at most one sudo block (PGDG repo + `apt install postgresql-16 postgresql-16-pgvector` + `sudo -u postgres createuser -s $USER`). No docker, no `usermod -aG docker`.

## Deferred follow-ups (post-v0.10)

These do not block the milestone; they're tracked here so they're not forgotten when v0.11 starts.

- **Manual fresh-VM acceptance test (operator-side).** The 2026-05-02 acceptance run simulated a fresh state via `XDG_CONFIG_HOME` and a temp secrets root, but didn't exercise: `apt install postgresql-16` against a host without Postgres, `sudo -u postgres createuser -s $USER`, `loginctl enable-linger`, or `systemctl --user start stevens-security` against a real user-session bus. The cloud sandbox available to remote agents can't run these (no sudo, no systemd-nspawn). Sol to run this manually on a clean VM or `systemd-nspawn` container at next opportunity; report any drift here under a "post-acceptance findings" subsection.
- **Native `signal-cli` install recipe.** `dev/compose.yaml` is the only remaining docker dependency, used only by `signal-cli-rest-api`. An apt + systemd user unit recipe to run `signal-cli` natively closes the gap and lets us drop docker from the repo entirely.
- **macOS launchd + Windows scheduled-task paths** for `bootstrap.systemd`. Currently `NotImplementedError` stubs. Land when there's a non-Linux operator to test against.
- **`stevens-core` PyPI publish.** Discussed in Open Questions below — consensus was "land with v0.11 plugins so the entry-point machinery has a real surface to wire into."
