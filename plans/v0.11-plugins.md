# Build Plan — v0.11-plugins

**Goal:** turn channels and Mortals into independently installable plugins discovered via Python entry points, with per-plugin manifests declaring capability scope, secrets, and system deps. Stevens core ships only the Pantheon, the loader, and the plugin runtime.

**Status:** queued — drafted 2026-05-02
**Predecessors:** v0.10-bootstrap (the install machinery this milestone extends).
**Charter refs:** STEVENS.md §2 Principles 12 (Pantheon/Mortals) and 13 (Plugins, not monoliths). Architecture: `docs/architecture/pantheon.md`, `docs/architecture/agent-isolation.md` §§3.6 (capability grant width) and 4 (Mortal lifecycle).

---

## Why this milestone exists

Today the channels (gmail, calendar, whatsapp-cloud, signal) live in `channels/<name>/` directories inside the main repo, and the Mortals (email_pm, installer) live in `agents/src/agents/<name>/`. Adding a channel or hiring a Mortal is a code change in core — even though their interfaces are already stable enough to live behind a manifest. This:

- Couples the release cadence of every channel to the core release.
- Means third parties can't ship adapters without touching core.
- Bloats core install footprint (you install the WhatsApp Cloud adapter even if you only use Gmail).
- Makes "uninstall channel X" non-trivial — there's no reverse of `git add`.
- Same problem for Mortals: there is no clean `stevens hire retire trip-planner` if `trip-planner` is in-tree code.

The fix is the standard Python plugin pattern: each channel and each Mortal is its own pip-installable package with a manifest declaring what it needs, discovered at startup via entry points.

---

## Scope

1. **Plugin manifest format** — `plugin.yaml` (or `pyproject.toml` table) declaring: name, kind (`channel` | `mortal`), capabilities required, channels required (for Mortals), secrets required, system deps, version, source URL, ownership.
2. **Entry-point discovery** — Python entry points under `stevens.channels` and `stevens.mortals` namespaces. Stevens scans these on startup; whatever's installed is what's available. The hardcoded `_CHANNELS` list in `channels_list.py` becomes a *registry of available* plugins (catalog), separate from *installed* (entry-point scan).
3. **`stevens channels`** subcommands: `list` (installed), `registry` (catalog of known channels), `install <name>` (pip install + bootstrap hook), `uninstall <name>` (revoke caps + drop schema + pip uninstall), `show <name>` (manifest + state).
4. **`stevens hire`** subcommands: `list`, `registry`, `install <name>` (install plugin Mortal), `spawn <spec>` (ad-hoc create from inline spec), `show <id>`, `retire <id>`, `pause <id>` / `resume <id>`.
5. **Per-Mortal Postgres schema** — each Mortal at hire time gets `mortal_<id>` schema. Migrations scoped to the Mortal run in that schema. On retire, the schema is archived (default) or dropped (with `--drop-data`).
6. **Capability grants visible per Mortal** — `stevens hire show <id>` prints the manifest's declared capability scope, the actually-granted scope (after Sol's approval), and the audit-tail tally of calls actually made. Lets Sol see drift between declared and used.
7. **Migrate existing in-tree code** to the plugin shape:
   - `channels/gmail/` → `stevens-channel-gmail` plugin (lives in `plugins/stevens-channel-gmail/` for now; separate repo in a later cleanup pass).
   - Same for `channels/calendar/`, `channels/whatsapp-cloud/`, `channels/signal/`.
   - `agents/src/agents/email_pm/` → `stevens-mortal-email-pm` plugin.
   - `agents/src/agents/installer/` → `stevens-mortal-installer` plugin.
   - The `channels/`, `agents/` directories are emptied or removed; core only ships the Pantheon (`security/`, `agents/src/agents/{web,pdf,janus}/` if those Pantheon members live in the agents tree, plus future memory + interface).
8. **Plugin runtime** — `stevens-runtime` daemon (a systemd user unit) that scans entry points, loads installed plugins, supervises their processes, and restarts them on failure. Replaces the per-service systemd units approach from v0.10 step 3 — one runtime supervises N plugins.

Out of scope:
- Public PyPI publishing of the per-channel plugins. They live in-tree under `plugins/` for v0.11; PyPI publishing happens when there's a real third-party need.
- A plugin marketplace. The `stevens channels registry` command reads from a single curated YAML in core for now; federated registries are a v2 thought.
- Sandboxing plugins beyond the existing capability/policy model. A plugin that gets approved capabilities is trusted with those capabilities; the security boundary is at Enkidu, not at the plugin process boundary.

## Plugin manifest spec (draft)

```yaml
# plugin.yaml — committed to the plugin's repo
name: gmail
kind: channel                       # or: mortal
display_name: Gmail
version: 1.0.0
source: https://github.com/stevens-ai/stevens-channel-gmail
maintainer: Sol

# Capabilities this plugin needs to function.
# For channels: capabilities the adapter calls.
# For Mortals: capabilities the agent's policy block declares.
capabilities:
  - gmail.send
  - gmail.read
  - gmail.draft

# Channels a Mortal needs (Mortal-only).
# Sol must have these channels installed before the Mortal can be hired.
channels:                           # Mortal-only field
  - gmail

# Sealed-store secrets the plugin expects, with prompts.
# Bootstrap hook checks these are present and runs the right onboarding.
secrets:
  - name: gmail.oauth_client.id
    prompt: "Google OAuth client ID"
    onboard_via: "stevens wizard google"
  - name: gmail.oauth_client.secret
    prompt: "Google OAuth client secret"
    onboard_via: "stevens wizard google"

# System dependencies (apt packages, brew formulas, …).
# If missing at install time, the installer Mortal proposes a plan to install.
system_deps:
  apt: []
  brew: []

# Bootstrap hook — Python entry point invoked on `stevens channels install <name>`.
# Receives a context object with sealed-store + db handles via Enkidu.
bootstrap: stevens_channel_gmail.bootstrap:install

# Adapter / agent entry point — what the runtime imports to start the plugin.
entry: stevens_channel_gmail.adapter:Adapter
```

The `pyproject.toml` of the plugin declares the entry-point side:

```toml
[project.entry-points."stevens.channels"]
gmail = "stevens_channel_gmail:manifest"
```

The runtime calls `manifest()` → `Manifest` object → which knows where the `plugin.yaml` lives.

## Steps

> `[ ]` todo · `[~]` in progress · `[x]` done.

### Step 1 — manifest schema + parser `[ ]`

`shared/src/shared/plugins/manifest.py`. Pydantic model for `plugin.yaml`. `load_manifest(package_name) → Manifest`. Validation: capabilities exist in the Pantheon's registry, channels in the available-channels list, secrets follow `<channel>.<account>.<key>` shape.

**Test:** load a hand-written gmail.yaml; valid case + each invalid case.

### Step 2 — entry-point discovery `[ ]`

`shared/src/shared/plugins/discovery.py`. `discover(kind: Literal['channel','mortal']) → list[InstalledPlugin]`. Wraps `importlib.metadata.entry_points`. Returns name + manifest + entry callable.

**Test:** install a fake plugin in a `tmp_path` venv, discover it.

### Step 3 — `stevens channels` CLI rewrite `[ ]`

Replace the hardcoded `_CHANNELS` list in `channels_list.py` with discovery output. `stevens channels list` shows installed (via discovery). `stevens channels registry` shows known catalog (a curated `resources/channels_registry.yaml`). `install / uninstall / show` wired through pip + bootstrap hook.

**Test:** `channels list` against a venv with no plugins → empty. After install of `stevens-channel-gmail`, lists it.

### Step 4 — `stevens hire` CLI `[ ]`

New top-level subcommand. `list`, `registry`, `install`, `spawn`, `show`, `retire`, `pause`, `resume`. Per-Mortal-id state in a `mortals` table (created by a new migration). `retire` calls Enkidu to revoke the policy block + tombstones the keypair.

**Test:** end-to-end on a fake mortal: install, spawn, show, retire — all the state transitions land in the `mortals` table and Enkidu's policy.

### Step 5 — per-Mortal Postgres schema `[ ]`

On hire: `CREATE SCHEMA IF NOT EXISTS mortal_<id>` + run the Mortal's own migrations under that schema (set `search_path` per session). On retire: `ALTER SCHEMA mortal_<id> RENAME TO archived_mortal_<id>_<timestamp>` (default) or `DROP SCHEMA … CASCADE` (with `--drop-data`).

**Test:** schema lifecycle in an integration test against real Postgres.

### Step 6 — runtime supervisor `[ ]`

`stevens_runtime` package. systemd user unit. Scans plugin entry points at startup, starts each as a subprocess, supervises with restart-on-failure backoff. Per-plugin logs go to `~/.local/state/stevens/logs/<name>.log`.

**Test:** start runtime with two fake plugins; kill one; verify restart. Stop runtime; verify all plugins die cleanly.

### Step 7 — migrate existing channels `[ ]`

Move each of `channels/{gmail,calendar,whatsapp-cloud,signal}/` into `plugins/stevens-channel-<name>/` with a `pyproject.toml` declaring entry points and a `plugin.yaml`. Each gets its own version, README, and bootstrap hook (the bootstrap hook codifies the existing `add_account.py` flow).

**Test:** for each channel, the pre-v0.11 onboarding flow still works through `stevens channels install <name>` instead of the old direct `python -m channel_adapter.add_account`.

### Step 8 — migrate existing Mortals `[ ]`

Same shape for `email_pm` and `installer`: extract to `plugins/stevens-mortal-<name>/`, declare manifest, register entry point. The empty `agents/src/agents/{email_pm,installer}/` directories get removed.

**Test:** `stevens hire install email-pm` reproduces the current Email PM behavior end-to-end.

### Step 9 — runbook + doc overhaul `[ ]`

Rewrite per-channel runbooks to lead with `stevens channels install <name>` instead of in-tree onboarding scripts. Add a "Hires" section to the master runbook covering `stevens hire install / spawn`.

**Test:** install Stevens fresh + install one channel + hire one Mortal following only the runbooks.

### Step 10 — acceptance `[ ]`

End-to-end on a clean install:

1. `uv run stevens bootstrap` (from v0.10).
2. `uv run stevens secrets init`.
3. `uv run stevens channels install gmail` → pip-installs the gmail plugin, runs bootstrap hook, prompts for OAuth.
4. `uv run stevens hire install email-pm` → installs the Mortal plugin, presents capability scope for approval, spawns the supervised process.
5. Send a test email. Email PM picks it up and drafts a reply through Enkidu.
6. `uv run stevens channels uninstall gmail` (after retiring email-pm) → removes plugin, revokes caps, removes schema. Stevens is back to a clean Pantheon-only state.

## Open questions

- Should Mortal manifests support *parameterization* on hire? (e.g. trip-planner takes `--for-trip "Tokyo Sept 2026"` at hire time, distinct from a separate trip-planner instance for a different trip.) Yes, almost certainly — but worth confirming the parameter shape before locking it in.
- How does a plugin declare it depends on another plugin (e.g. a Mortal needs a channel that isn't installed)? Manifest's `channels:` list + a preflight check at hire time that errors with "install <channel> first" — sufficient for v0.11.
- Plugin signing / chain of trust? Out of scope for v0.11. Sol installs from his own trusted sources; signing matters when there's a marketplace.
