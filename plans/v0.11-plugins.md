## Why this milestone exists

Today the channel adapters (gmail, calendar, whatsapp-cloud, signal) live in `channels/<name>/` directories inside the main repo, and the Mortals (email_pm, installer) live in `agents/src/agents/<name>/`. Adding a new external integration or hiring a Mortal is a code change in core — even though their interfaces are already stable enough to live behind a manifest. This:

- Couples the release cadence of every integration to the core release.
- Means third parties can't ship integrations without touching core.
- Bloats core install footprint (you install the WhatsApp Cloud adapter even if you only use Gmail).
- Makes "uninstall integration X" non-trivial — there's no reverse of `git add`.
- Same problem for Mortals: there's no clean `demiurge hire retire trip-planner` if `trip-planner` is in-tree code.
- The "channel" abstraction also doesn't fit emerging integrations cleanly. A stock-price API isn't a channel — there's nothing inbound. An image generator isn't a channel either. A polling RSS reader is somewhere between. Whatever we call it, an external-world integration is shaped by *which mechanisms it uses* (webhook, listener, polling, request-based), not by whether it's "communication."

The fix is the standard Python plugin pattern + a unified vocabulary. Each integration is a **power** — a pip-installable package with a manifest declaring its mechanism mix, capabilities, secrets, and bootstrap. Mortals are similar but task-scoped.

---

## Vocabulary

- **Power** — any external-world integration. Replaces the channel-only framing. A power's manifest declares one or more **modes**:

| Mode | Direction | Trigger | Example |
|---|---|---|---|
| **Webhook** | inbound | remote pushes to us (HTTPS endpoint) | Gmail Pub/Sub, WhatsApp Cloud, Stripe |
| **Listener** | inbound | we hold a long-lived outbound connection | IMAP IDLE, Signal websocket, MQTT |
| **Polling** | inbound | we hit them on a schedule | RSS feed, stock price |
| **Request-based** | outbound | agent calls on demand | image generator, web search |

A power can combine modes (Gmail = webhook + request-based; Signal = listener + request-based; image generator = request-based-only; RSS = polling-only).

- **Pantheon members** stay where they are: Enkidu, Arachne, Sphinx, Janus shipped; Hephaestus + Hades planned for this milestone; Mnemosyne + Iris planned for v0.12. Pantheon members are not powers — they're the substrate powers and Mortals depend on.
- **Mortals** stay as defined: task-scoped agents, can be hired and retired, declare their dependencies on powers + capabilities in their manifest.

---

## Scope

1. **Plugin manifest format** — `plugin.yaml` (or `pyproject.toml` table) declaring: name, kind (`power` | `mortal`), modes (powers only — list of webhook/listener/polling/request-based), runtime block (per-mode runtime details), capabilities required, powers required (for Mortals), secrets required, system deps, version, source URL, ownership.
2. **Entry-point discovery** — Python entry points under `demiurge.powers` and `demiurge.mortals` namespaces. Demiurge scans these on startup; whatever's installed is what's available. The hardcoded `_CHANNELS` list in `channels_list.py` becomes a *registry of available* plugins (catalog), separate from *installed* (entry-point scan).
3. **`demiurge powers`** subcommands: `list` (installed), `registry` (curated catalog of known powers), `install <name>` (pip install + Hephaestus bootstrap hook), `uninstall <name>` (Hades teardown — revoke caps, drop schema, pip uninstall), `show <name>` (manifest + state). The legacy `demiurge channels …` aliases remain functional but are documented as deprecated.
4. **`demiurge hire`** subcommands: `list`, `registry`, `install <name>` (install plugin Mortal — Hephaestus forge), `spawn <spec>` (ad-hoc Mortal from inline spec — Hephaestus forge), `show <id>`, `retire <id>` (Hades archive), `pause <id>` / `resume <id>`.
5. **Per-Mortal Postgres schema** — each Mortal at hire time gets `mortal_<id>` schema. Migrations scoped to the Mortal run in that schema. On retire (Hades): the schema is archived (default) or dropped (with `--drop-data`).
6. **Capability grants visible per Mortal + per power** — `demiurge hire show <id>` and `demiurge powers show <name>` print the manifest's declared capability scope, the actually-granted scope (after Sol's approval), and the audit-tail tally of calls actually made. Lets Sol see drift between declared and used.
7. **Migrate existing in-tree code** to the plugin shape:
   - `channels/gmail/` → `demiurge-power-gmail` plugin (lives in `plugins/demiurge-power-gmail/` for now; separate repo in a later cleanup pass).
   - Same for `channels/calendar/`, `channels/whatsapp-cloud/`, `channels/signal/`.
   - `agents/src/agents/email_pm/` → `demiurge-mortal-email-pm` plugin.
   - `agents/src/agents/installer/` → `demiurge-mortal-installer` plugin.
   - The `channels/`, `agents/` directories are emptied or removed; core only ships the Pantheon (`security/`, `agents/src/agents/{web,pdf,janus}/` if those Pantheon members live in the agents tree, plus future memory + interface).
8. **Hephaestus** (`forge` Pantheon member) — concrete implementation of the create-side lifecycle executor. Reads a manifest, registers capabilities with Enkidu, generates the runtime artifact per the manifest's `runtime:` block (no unit for request-based-only powers; `.service` for webhook/listener; `.service`+`.timer` for polling), wires bus subscriptions, records the apotheosis hook for any future promotion. Owns Apotheosis, Succession (forge side), Binding (forge side).
9. **Hades** (`underworld` Pantheon member) — concrete implementation of the end-side lifecycle executor. Tears down the runtime, revokes capabilities, archives the audit trail and last state to `archived_mortal_<id>_<ts>` schema (or `archived_power_<name>_<ts>`), frees secrets back to Enkidu. Owns Fading, Exile, Ragnarök, Succession (sever side), Binding (archival side).
10. **Plugin runtime supervisor** — `demiurge_runtime` daemon (a systemd user unit). Scans plugin entry points at startup, starts each reactive (webhook/listener/polling) plugin as a subprocess per its `runtime:` block, supervises with restart-on-failure backoff. Replaces the hardcoded 6-unit catalog from v0.10 step 3 — Hephaestus generates the units from manifests; runtime supervises them.

Out of scope:
- Public PyPI publishing of the per-power plugins. They live in-tree under `plugins/` for v0.11; PyPI publishing happens when there's a real third-party need.
- A federated plugin marketplace. The `demiurge powers registry` command reads from a single curated YAML in core for now.
- Sandboxing plugins beyond the existing capability/policy model. A plugin that gets approved capabilities is trusted with those capabilities; the security boundary is at Enkidu, not at the plugin process boundary.

## Plugin manifest spec (draft)

```yaml
# plugin.yaml — committed to the plugin's repo
name: gmail
kind: power                         # or: mortal
display_name: Gmail
version: 1.0.0
source: https://github.com/<org>/demiurge-power-gmail
maintainer: Sol

# Modes (powers only) — which mechanism(s) this power uses.
# Hephaestus reads this to know what runtime artifact to generate.
modes:
  - webhook
  - request-based

# Per-mode runtime details (powers only).
# Each declared mode needs a corresponding runtime block.
runtime:
  webhook:
    path: /gmail/push                 # URL path on the FastAPI mux
    port: 8080                        # local port the listener binds to
    handler: demiurge_power_gmail.adapter:webhook_handler
  # listener:
  #   command: demiurge_power_signal.adapter:run_listener
  #   restart: on-failure
  # polling:
  #   command: demiurge_power_rss.fetch:run_once
  #   interval: 1h
  # (request-based has no runtime block — it just registers capabilities)

# Capabilities this plugin needs to function.
# For powers: capabilities the adapter calls.
# For Mortals: capabilities the agent's policy block declares.
capabilities:
  - gmail.send
  - gmail.read
  - gmail.draft

# Powers a Mortal needs (Mortal-only).
# Sol must have these powers installed before the Mortal can be hired.
powers:                             # Mortal-only field (was: channels)
  - gmail

# Sealed-store secrets the plugin expects, with prompts.
# Bootstrap hook checks these are present and runs the right onboarding.
secrets:
  - name: gmail.oauth_client.id
    prompt: "Google OAuth client ID"
    onboard_via: "demiurge wizard google"
  - name: gmail.oauth_client.secret
    prompt: "Google OAuth client secret"
    onboard_via: "demiurge wizard google"

# System dependencies (apt packages, brew formulas, …).
# If missing at install time, the installer Mortal proposes a plan to install.
system_deps:
  apt: []
  brew: []

# Bootstrap hook — Python entry point invoked on `demiurge powers install <name>`.
# Receives a context object with sealed-store + db handles via Enkidu.
bootstrap: demiurge_power_gmail.bootstrap:install
```

The `pyproject.toml` of the plugin declares the entry-point side:

```toml
[project.entry-points."demiurge.powers"]
gmail = "demiurge_power_gmail:manifest"
```

The runtime calls `manifest()` → `Manifest` object → which knows where the `plugin.yaml` lives.

### Step 1 — manifest schema + parser `[x]` (2026-05-02)

## Architectural decisions locked 2026-05-03 (apply throughout)

These came out of the Hephaestus design discussion and bind the rest of v0.11.

1. **Four creature kinds** — Mortal (full agency), Beast (model-driven, no agency), Automaton (deterministic, no LLM), Angel (god-extension, opaque). Future fifth: Prophet, reserved. See DEMIURGE.md §1.1 + docs/architecture/gods.md.
2. **No journal, no scratchpad, no diary in Creature code.** Universal tools shrink to `think(text)` (speak aloud for the audit angel) + `mortal.return(result)` only. Memory recall is a blessed tool `tools.memory.recall(query)` (Mnemosyne stub before v0.13).
3. **Single observation feed per Creature** at `~/.local/state/demiurge/creatures/<id>/events.jsonl`, mode 0640, owned by Enkidu's uid. Stable envelope: `creature_id`, `event_id` (UUIDv7), `ts` (ISO8601 µs), `kind`, `correlation_id`, `data`. Single time source, single ID namespace. Cross-angel join is `JOIN ON event_id`.
4. **Audit angel mandatory.** Enkidu always commissions an audit angel for every Creature. v0.11 ships it as in-process Enkidu code projecting the feed into the existing tamper-evident audit log. v0.13 promotes it to a real out-of-process angel alongside Mnemosyne's memory angel.
5. **Mortals do not summon.** No `mortal.summon` tool. No `can_summon` manifest field. Creature creation is a god-level act. v0.11: only Sol spawns (via `demiurge hire`). v0.12+: Mortals request from Zeus (`zeus.request_spawn`-shaped blessed tool); Iris is in the path *only* when the request originates from Sol.
6. **Permissions immutable.** A Creature's blessings are set at forge time and don't mutate for its lifetime. Change = retire + re-forge. Static authority graph; no dynamic permission lending. Mortals working *with* each other within their existing capabilities (bus events, calling existing Beasts/Automatons) is fine.
7. **Iris ≠ Zeus ≠ Demiurge.** Iris is Sol's personal UI (preferences, modality, notification routing). Zeus is the chairman (multi-god dispatch + judgment). Demiurge is the substrate (bootstrap, supervisor, lifecycle). All three planned post-v0.11.
8. **Cross-god access:** reads allowed (any god can read another's substrate when investigating); writes forbidden (each god is the only writer to its own substrate; integrity follows from single-writer rule).

#### Step 3c — Tool routing + dispatch `[x]` (2026-05-03)

Prefix → god mapping (`gmail.*` → Enkidu; `web.*` → Arachne; `pdf.*` → Sphinx; `browser.*` → Janus; `memory.*` → Mnemosyne stub; `say.*` → Iris stub; `zeus.*` → Zeus stub). `forge_blessings(manifest) → BlessedToolRegistry`. ## Open questions

- **Mortal manifests on hire — parameterization.** Trip-planner takes `--for-trip "Tokyo Sept 2026"` at hire time, distinct from another trip-planner instance for a different trip. Yes, almost certainly worth supporting — confirm parameter shape (Pydantic model declared in the manifest's `params:` field, validated against the inline spec at `spawn` time?) before locking it in.
- **Plugin → plugin dependencies.** A Mortal manifest's `powers:` list + a preflight check at hire time that errors with "install <power> first" — sufficient for v0.11. Plugin → plugin Mortal dependencies (one Mortal needs another Mortal already running) — defer to v0.11.x if it comes up.
- **Plugin signing / chain of trust.** Out of scope for v0.11. Sol installs from his own trusted sources; signing matters when there's a marketplace.
- **Per-power webhook port allocation.** Today each adapter binds its own port (8080, 8082, 8083). Plugins should declare the port they want; Hephaestus needs to detect collisions and either remap or error. Worth deciding if v0.11 ships a single FastAPI mux that hosts all webhook routes (cleaner) vs preserving per-plugin port binding (simpler). Lean: single mux, but confirm before locking.
