# Build Plan — v0.12-pantheon-expansion

**Goal:** add **Mnemosyne** (memory/context) and **Iris** (UI/persona) to the Pantheon. These are the two long-anticipated core agents that Mortals will depend on for memory and for any user-facing dialogue.

**Status:** queued — drafted 2026-05-02
**Predecessors:** v0.11-plugins (because both agents need the plugin runtime + the per-Mortal isolation model in order to be usable surface for Mortals).
**Charter refs:** STEVENS.md §1.1 (Pantheon names), §4.2 (Pantheon vs Mortals — Mnemosyne and Iris listed as planned), §4.5 (Context and memory — Mnemosyne's domain). Architecture: `docs/architecture/pantheon.md`. Memory deeper writeup deferred to a future `MEMORY.md` charter doc.

---

## Why this milestone exists

Today every Mortal that needs to remember something across sessions has to invent its own storage shape. Every Mortal that wants to talk to Sol has to invent its own UI. Both are *shared, repeated responsibilities* — exactly the criterion in `pantheon.md` for promotion to the Pantheon. Doing this once, well, and centrally, beats doing it twelve times across twelve Mortals.

Memory and UI being core also unblocks the longer-term vision: Mortals can be small and focused because they delegate "remember this," "ask Sol if it's OK to send X," "draft me a clarifying question," and "show this in tomorrow's briefing" upward to Mnemosyne and Iris.

This is also where the deferred PRD §3.4 "Context Management Agent" and "UI Agent (Stevens persona)" land for real, with code identifiers and named characters.

## Scope

### Mnemosyne (memory)

1. **Code identifier:** `memory`. **Display name:** Mnemosyne (Greek titaness of memory, mother of the Muses).
2. **Capability surface** exposed via Enkidu (so Mortals never touch storage directly):
   - `memory.note(scope, key, value, ttl?, redact_rules?)` — write a structured memory entry under a scope (per-Mortal namespace by default).
   - `memory.recall(scope, query, k=10, filters?)` — vector + structured search.
   - `memory.forget(scope, key | query)` — explicit deletion (audit-logged).
   - `memory.export(scope)` — operator-only; returns the full memory of a scope for review.
3. **Storage:** Postgres + pgvector (chosen in the 2026-05-02 discussion as the single-DB solution). Embeddings via the local model (no cloud egress for memory writes by default).
4. **Scopes:** every Mortal has its own namespace; promotion of a memory to a *shared* scope requires an explicit Mortal-side capability call which is gated by Sol's approval, not silent.
5. **Redaction-before-recall:** memories tagged with `redact_rules` (e.g. "PII", "credentials-adjacent") get redacted on recall unless the recalling caller has the matching capability. The default Mortal grant does *not* include the unredacted-recall capability.
6. **Forgetting policy:** Sol-configurable per-scope (default: keep forever for personal scopes, 90-day TTL for ephemeral Mortals).

### Iris (interface / persona)

1. **Code identifier:** `interface`. **Display name:** Iris (Greek messenger goddess, rainbow bridge between gods and mortals — perfect fit for the UI surface).
2. **Capability surface:**
   - `interface.notify(level, channel, message, attachments?)` — sends a message to Sol over the chosen channel (CLI banner, email, Telegram, etc.). Replaces Mortals doing their own outbound user-facing communication.
   - `interface.ask(channel, question, options?, ttl?)` — synchronous ask-Sol pattern; returns Sol's answer via the chosen channel. Used by Mortals for clarifying questions and approval requests.
   - `interface.brief(scope, content)` — schedules content for Sol's daily briefing.
3. **Persona enforcement:** all outbound user-facing text is composed by Iris in the Stevens voice (loaded from `SOUL.md` per `agent-isolation.md` §11). Mortals provide *intent* + structured payload; Iris does the wording. This guarantees Sol always hears one voice regardless of which Mortal originated the message.
4. **Channels Iris can use:** any installed channel that has a `send.user_facing` capability. Iris owns the channel-selection logic (e.g. "this is high-priority — interrupt; this is informational — tomorrow's briefing").
5. **Approval surface:** Iris is the front-end for Enkidu's approvals queue. When a Mortal triggers an approval-gated capability, Iris is responsible for surfacing it to Sol via the right channel and recording the response.

### Cross-cutting

1. **Migration of existing Mortal-internal memory** — anything Email PM or installer currently stashes ad-hoc gets ported to `memory.note` calls. This is a small migration; both have minimal state today.
2. **Existing CLI banners that pretend to be Stevens** — `stevens audit tail`, `stevens doctor` outputs etc. — get rewritten to actually go through Iris when the user-facing context calls for it, vs. raw operator output (which stays raw).

Out of scope:
- Cross-host memory sharing (Stevens runs on one box for now).
- Persona-tuning UI (you edit `SOUL.md` directly; a tuner agent is a future Mortal, not a Pantheon concern).
- Voice / audio channels for Iris (text only for v0.12).

## Steps

> `[ ]` todo · `[~]` in progress · `[x]` done.

### Step 1 — Mnemosyne skeleton + storage `[ ]`

Pantheon member at `agents/src/agents/memory/`. Postgres tables: `memory_entries` (scope, key, value, embedding, metadata, ttl, redact_rules) + `memory_scopes` (registry of valid scopes per Mortal). Migration adds the tables + the pgvector extension (with a clear error if pgvector isn't installed, pointing at the install command).

**Test:** write + recall a few entries; vector similarity lands the expected order; TTL expiration works.

### Step 2 — Mnemosyne capabilities `[ ]`

`memory.note`, `memory.recall`, `memory.forget`, `memory.export` registered in the capability registry. Default policy: every Mortal automatically gets `memory.note` and `memory.recall` scoped to its own namespace; `memory.forget` requires explicit grant; `memory.export` is operator-only.

**Test:** policy enforcement — Mortal A cannot recall from Mortal B's scope. `memory.export` denied to a Mortal.

### Step 3 — Mnemosyne redaction `[ ]`

Redact-on-recall middleware. A memory tagged `redact_rules: [pii]` returns with PII fields masked unless the caller has `memory.recall.unredacted:pii`. Audit trail records both the redacted and unredacted access patterns.

**Test:** PII pattern (email addresses, phone numbers) gets masked for default callers; unmasked for an explicitly-granted caller.

### Step 4 — Iris skeleton + persona load `[ ]`

Pantheon member at `agents/src/agents/interface/`. Loads `SOUL.md` via the existing `safe_load_user_markdown`. Compose pipeline: `intent + payload` → LLM call with persona-as-system-prompt → outbound text.

**Test:** `interface.notify(level=info, channel=cli, "test")` round-trips through the persona pipeline and emits Stevens-voiced text.

### Step 5 — Iris capabilities `[ ]`

`interface.notify`, `interface.ask`, `interface.brief` registered. Default policy: every Mortal gets `interface.notify` and `interface.ask` (the most basic communication surface); `interface.brief` requires explicit grant.

**Test:** Mortal calls `interface.ask`, Iris routes to the configured channel, Sol's response comes back through the same capability call.

### Step 6 — Iris ↔ approvals integration `[ ]`

Approvals queue (currently surfaced via `stevens approval list`) gains an Iris path: pending approvals are pushed to Sol via the appropriate channel automatically. Sol's response (via that channel) flows back into Enkidu's approval-grant API.

**Test:** trigger an approval-gated capability call; verify Iris surfaces it to the chosen channel; respond; verify the call replays and succeeds.

### Step 7 — migrate ad-hoc Mortal memory to Mnemosyne `[ ]`

Audit Email PM and installer for any local state they're keeping; replace with `memory.note` / `memory.recall` calls. Small change.

**Test:** existing Email PM behavior still works after migration.

### Step 8 — STEVENS.md + agent-isolation update `[ ]`

Mark Mnemosyne and Iris as **shipped** in §1.1 and §4.2. Add Mnemosyne and Iris to the Pantheon members listed in `agent-isolation.md` §3.6. If a real `MEMORY.md` charter doc lands as part of this milestone, link from STEVENS.md §4.5.

### Step 9 — acceptance `[ ]`

End-to-end:

1. Hire a fresh Mortal (e.g. `trip-planner`) via the v0.11 `stevens hire install` flow.
2. Mortal calls `memory.note` to record a fact. Verify entry lands in `mortal_trip_planner` scope.
3. Mortal calls `interface.ask("when do you fly?")`. Verify Sol receives the question via his chosen channel, in Stevens voice.
4. Mortal calls `memory.recall` and uses the answer.
5. Retire the Mortal. Verify memory namespace archives cleanly; subsequent `memory.recall` from a new Mortal in the same scope returns nothing.

## Open questions

- Iris persona file: do we use a single `SOUL.md` for the Stevens voice, or layer in per-Mortal voice tweaks (a Mortal handling kid-related tasks might want a different register)? Suggest: single `SOUL.md` for v0.12; per-Mortal overlays if we end up needing them.
- Memory scope hierarchy: are scopes flat (one per Mortal) or nested (Mortal → topic → entry)? Flat for v0.12; nested if it turns out we need it.
- Briefing cadence: who decides what goes in tomorrow's briefing — Iris polling Mortals' `interface.brief` calls, or each Mortal pushing? Iris-pulled is the right shape (Iris owns the schedule), but the API is push-based with Iris doing the dedup + sequencing.
- Embedding model choice: local Qwen3-embedding default vs the larger English-language models. Likely local; depends on what we have available on the 3090.
