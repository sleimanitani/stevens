# Build Plan ‚Äî v0.2-skills: shared skills layer (tools + playbooks)

**Goal:** ship `skills/` as the substrate every future agent is built on. Tools and playbooks are first-class, separate, and reused across agents. Email PM moves to use it end-to-end. PDF reader is the canonical first shared tool and is acceptance-gating.

**Status:** in-progress ‚Äî started 2026-04-30
**Charter refs:** `STEVENS.md` (reuse over regenerate), `CLAUDE_skills_layer.md` (canonical spec), `CLAUDE_openclaw_evaluation.md` (frontmatter format + borrow-don't-depend posture).
**Predecessors:** `v0.1-sec` (`133dd78`), `v0.1.6-ergonomics` (`9b32865`).
**Blocks:** Email PM live run; subject agents (Berwyn, etc.); the interface agent's SOUL.md (v0.2.x+).

## Scope

In this milestone:

1. `skills/` package + workspace registration.
2. `skills.registry` lookup API + runtime wiring.
3. Playbook loader with **agentskills.io-compatible** frontmatter (top-level `name/description/version/author/license`, our extensions under `metadata`).
4. Trigger-match retrieval (v1) with stable interface for v2 (pgvector).
5. Migration `004_skill_proposals.sql` + table.
6. `propose_skill` shared tool ‚Äî agent-callable, drops to DB + filesystem; agent can't use its own proposal.
7. **PDF reader** at `skills/src/skills/tools/pdf/read_pdf.py` (acceptance gate).
8. `scripts/review_skills.py` ‚Äî `list / show / approve / reject`.
9. Email PM rewired to use the registry; six starter playbooks shipped through the propose‚Üíapprove flow.
10. Security-side audit: confirm Enkidu doesn't migrate to `skills/`. Document boundary in STEVENS.md.
11. DEVELOPMENT.md sections: adding tools, adding playbooks, reviewing proposals.

Out of scope (deferred to a v2 of the skills layer):

- `skills/distiller.py` (autonomous Langfuse-trace mining).
- pgvector retrieval.
- Runtime enforcement of `safety_max` (schema fields ship; enforcement waits until a `destructive` tool exists).
- Browser Harness / OpenClaw channel adapters as new code (demand-driven; port when needed).

### Step 3 ‚Äî Playbook loader (agentskills.io-compatible frontmatter) [x]

### Step 4 ‚Äî Trigger-match retrieval (v1) [x]

`get_playbooks_for(agent, event) -> list[Playbook]`:
1. `agent_name in playbook.metadata.applies_to_agents` (or list is empty/missing ‚Üí applies to all).
2. `event.topic` matches one of `applies_to_topics` patterns (NATS-style: `*` matches one segment, `>` matches the rest ‚Äî but for v1 we just use `fnmatchcase` on the full topic since our events are flat).
3. At least one `triggers[].regex` matches the event's content (search in `subject` + `snippet` + `body` if present).
4. Status must be `active`.

Cap: 3 playbooks (lower than the doc's 5 ‚Äî pragmatic for Qwen3-30B's context window). Make the cap configurable via env var `STEVENS_MAX_PLAYBOOKS`.

**Files:**
- `skills/src/skills/retrieval.py` ‚Äî `get_playbooks_for(agent_name, event, *, max_playbooks=3) -> list[Playbook]`. Pure function over a loaded `Registry`.
- `skills/tests/test_retrieval.py` ‚Äî 6 tests: agent-scope match, topic-pattern match, trigger-regex match, deprecated playbook excluded, cap respected, status filter.

**Stable interface:** the public signature is fixed so a v2 pgvector implementation drops in by replacing the body ‚Äî no caller changes.

### Step 7 ‚Äî PDF reader (acceptance gate) [x]

`read_pdf` is the canonical first shared tool. Cross-page table merging is the non-trivial bit.

**Files:**
- `skills/src/skills/tools/pdf/__init__.py`
- `skills/src/skills/tools/pdf/read_pdf.py` ‚Äî `TOOL_METADATA` dict, `ReadPDFInput` pydantic schema, `_read_pdf()` impl, `build_tool() -> StructuredTool`. Helper `_merge_cross_page_tables(tables_per_page) -> tables`.
- `skills/tests/test_read_pdf.py` ‚Äî at least four PDFs:
  1. Text-only (extract text + zero tables).
  2. PDF with table spanning two pages ‚Üí one merged table out.
  3. Encrypted PDF ‚Üí structured error.
  4. Scanned PDF ‚Üí OCR fallback (skipped if `tesseract` binary absent; manual install note in DEVELOPMENT.md).
- `skills/tests/fixtures/` ‚Äî small PDFs (text-only, 2-page-table, encrypted, scanned-1page). Generated programmatically where possible (`reportlab` is a dev-only dep just for fixture generation; if too heavy, check in pre-built PDFs).

**Cross-page table merge rule:** if page N's last table and page N+1's first table have the same column count AND the column count > 1 AND page N+1's first row's cells don't look like headers (not all-caps, no obvious header markers), merge. Conservative ‚Äî better to leave two tables than mistakenly merge unrelated ones.

**Acceptance gating:** Steps 1‚Äì8 are not "done" until `read_pdf` passes all four test PDFs (or three with OCR skip if no tesseract).

### Step 8 ‚Äî `scripts/review_skills.py` [x]

Operator CLI to triage proposals.

**Subcommands:**
- `list` ‚Äî prints pending proposals (id ¬∑ kind ¬∑ title ¬∑ proposing_agent ¬∑ proposed_at).

- `reject <id> --reason "..."` ‚Äî marks rejected, leaves the file in `proposed/` (or moves to `proposed/rejected/<id>/`).

**Files:**
- `scripts/review_skills.py` ‚Äî argparse-based, no new deps.
- `scripts/tests/test_review_skills.py` ‚Äî 4 tests: list shows pending, approve moves file + updates yaml + flips status, reject flips status + leaves file, unknown id errors.

## Risks

- **Playbook prompt bloat blowing the context window.** Cap=3 + measure. Documented in `retrieval.py`. Bump to 5 only if measurements support it.
- **Trigger regex over-firing.** Each playbook proposal must include three sample inputs the trigger should match and one it should NOT. Reviewed by Sol.
- **Self-proposal/self-approval loop.** When Sol writes the six starter playbooks, he'll be both proposer and approver. The audit trail records both roles; the discipline is reading every line before approving. Not a violation.
- **Acceptance gate needs tesseract.** `apt install tesseract-ocr` is a one-line dev-machine setup. If absent, the scanned-PDF test skips with a clear marker. Three other PDF tests still hold the gate.
- **`skill_proposals` migration drift.** Migration 004 needs to be applied before `propose_skill` works. Document in the runbook + add a `stevens doctor` check in a follow-up (not this milestone).
- **`pdfplumber` cross-page tables are heuristic.** No library does this reliably; our merger has a conservative bias. Edge cases will surface; treat them as bugs against `read_pdf`, not against the architecture.
