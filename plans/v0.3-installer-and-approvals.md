# Build Plan — v0.3-installer-and-approvals

**Goal:** ship the **approvals primitive** (standing + per-call, orthogonal predicates) and the **installer agent** built on top of it. Tesseract install is the integration test. Lays the substrate for every future approval-gated capability (payments, credential rotation, autonomous-send).

**Status:** in-progress — started 2026-04-30
**Charter refs:** STEVENS.md §2 Principle 11, §3.13. Specs: `docs/architecture/agent-isolation.md`, `docs/protocols/approvals.md`, `docs/protocols/privileged-execution.md`.
**Predecessors:** v0.1-sec (`133dd78`), v0.1.6-ergonomics (`9b32865`), v0.2-skills (`d03a547`).
**Blocks:** real OCR (the skipped scanned-PDF test); future payment authorization, credential rotation, autonomous-send.

## Scope

In this milestone:

1. Migrations 005-008: `standing_approvals`, `approval_requests`, `install_plans`, `environment_packages`.
2. Approvals primitive: in-memory orthogonal-predicate matcher + per-call queue + lifecycle.
3. Policy evaluator extension: `requires_approval` + `rationale_required` flags.
4. Capability surface: `system.read_environment`, `system.plan_install`, `system.execute_privileged`, `system.write_inventory`.
5. `apt` mechanism module (validator + executor + structural health-check + rollback).
6. Installer agent at `agents/src/agents/installer/`.
7. CLI: `stevens approval [standing] {list/grant/approve/reject/revoke}`, `stevens dep {ensure/list/remove}`.
8. End-to-end mocked test exercising the full flow without real sudo.

Out of scope:
- pip / git / opt_dir / container mechanisms (apt covers tesseract, ffmpeg, poppler, etc.).
- Dep-graph resolver (operator runs `dep ensure` per dep).
- Web fetch + search (`v0.3.1-web`).
- Real Docker compose run with privileged container (declared manual; mocked path is the unit test).
- UI agent integration with installer (v0.4+).

### Step 4 — Per-call approval queue + dispatch integration [x]

When the policy evaluator returns `REQUIRES_APPROVAL`, the dispatcher checks the standing matcher first; on miss, it enqueues an `approval_requests` row, returns `BLOCKED` to the caller, and (eventually) replays the envelope when Sol approves.

**Files:**
- `security/src/stevens_security/approvals/queue.py` — `enqueue_request(envelope, rationale) -> request_id`, `decide(request_id, status, notes) -> Outcome`, `replay_approved() -> generator of envelopes ready to execute`.
- `security/src/stevens_security/dispatch.py` — extended to: (a) consult matcher on `REQUIRES_APPROVAL`, (b) on miss, write to queue and return BLOCKED, (c) on standing match, execute and audit with `approval_via: standing/<id>`.
- `security/src/stevens_security/audit.py` — `AuditEntry` gains `approval_via` and `approval_request_id` optional fields.
- `security/tests/test_approval_dispatch.py` — 8 tests: match-via-standing → silent execute, no-match → BLOCKED + queue row, approve → replay, reject → DENIED, BLOCKED audit, replay idempotency.

**Replay semantics:** the per-call approval CLI calls a "release" function in Enkidu that replays the queued envelope through the dispatch pipeline (skipping the policy gate for that exact request_id, since it's already approved). The replayed call is signed by Enkidu's internal token, audited as `approval_via: per_call/<req_id>`.

### Step 6 — `system.*` capabilities [x]

Four capabilities per `docs/protocols/privileged-execution.md`.

**Files:**
- `security/src/stevens_security/capabilities/system.py` — implements:
  - `system.read_environment` — narrow queries (os_release, package_manager, dpkg_status[name], opt_dirs[pattern]). Reads only; not approval-gated.
  - `system.plan_install` — calls mechanism's validator; on success, writes a row to `install_plans` with 30-min `expires_at`, returns `plan_id`.
  - `system.execute_privileged` — looks up plan, re-validates, **delegates approval check to the dispatcher** (via the matcher → queue path), on approval runs the executor in a privileged subprocess, runs health check, writes inventory, audits.
  - `system.write_inventory` — append-only inventory writes; sets `caller` from verified caller name; not approval-gated.
- `security/tests/test_capability_system.py` — 10+ tests, all with mocked `subprocess.run` so no real apt invocation.

**Subprocess safety:**
- `subprocess.run(argv, shell=False, env=…, timeout=…, capture_output=True)`.
- argv built strictly from validated plan.
- env explicitly set (no inherited env).
- on timeout: kill, run rollback, audit `health_check_failed: timeout`.

**Mocked-subprocess test design:** the mechanism's executor is a pure data structure; the actual `subprocess.run` is invoked from the capability handler. We monkeypatch it in tests to record argv/env/timeout, return preset stdout/stderr/exitcode. Real execution requires sudo + Docker compose; declared manual.

### Step 7 — Installer agent [x]

Narrow agent; no LLM logic in v0.3 (its job is deterministic). Could grow LLM reasoning in v0.3.1 if planning gets non-trivial.

**Files:**
- `agents/src/agents/installer/__init__.py`
- `agents/src/agents/installer/agent.py` — orchestrates handle(event):
  1. Receives `system.dep.requested.<name>`.
  2. Calls `system.read_environment` (cap).
  3. Builds a plan via local `plan_builder.build_apt_plan(name, env)` (in-agent skill).
  4. Calls `system.plan_install(plan)` (cap) → plan_id.
  5. Calls `system.execute_privileged(plan_id, rationale)` (cap, approval-gated).
  6. On `BLOCKED` → publishes `system.dep.awaiting_approval.<name>` event with the request_id; the operator-facing CLI `stevens dep ensure --wait` polls or subscribes.
  7. On `ok` → publishes `system.dep.installed.<name>`.
  8. On `deny` / failure → publishes `system.dep.failed.<name>` with reason.
- `agents/src/agents/installer/plan_builder.py` — pure logic: takes (package_name, env_snapshot) → returns plan dict.
- `agents/src/agents/registry.yaml` — adds installer entry with subscribes `["system.dep.requested.*"]`.
- `agents/tests/test_installer_agent.py` — 6 tests with mocked SecurityClient: happy path → installed event, BLOCKED → awaiting_approval event, denied → failed event, plan validation error → failed event, env-incompatible → failed event with reason.

**Restricted scope:**
- No skills tools other than `installer.compat_check` (a local skill that wraps `system.read_environment`).
- No bus subscriptions other than `system.dep.requested.*`.
- No DB access except via the system.write_inventory capability.

## Risks

- **Replay correctness.** When approving a per-call request, Enkidu must replay the *original signed envelope* — not synthesize a new one. The agent's signature on the original is what authorizes the operation; Enkidu approving doesn't grant the agent's privileges, only releases the gate. Stored as a JSONB blob in `approval_requests.full_envelope`; replayed verbatim through the dispatch pipeline with a "skip-policy" flag for the matching request_id. Test explicitly: replayed envelope has the agent's signature, not Enkidu's.
- **Standing-approval cache invalidation race.** Between Sol running `revoke` and the next call, there's a window where Enkidu's in-memory cache still says "approved." Mitigation: `revoke` writes the DB then sends a SIGHUP-equivalent signal Enkidu acts on synchronously; if Enkidu is unreachable, revoke still writes the DB (so next boot loads correctly). Acceptable race for v0.3; tightened in v1.0 if needed.
- **apt non-determinism.** apt's exit codes are sometimes 0 on partial failures; the executor leans on the **structural health check** (dpkg-query) to confirm success rather than trusting the exit code alone.
- **Plan expiry vs execution time.** A plan expires 30 min after `plan_install`. If the per-call approval takes longer than 30 min, the call fails at execute_privileged time. The CLI surfaces this to the operator and the agent can re-plan. Documented behavior, not a bug.
- **Inventory reconciliation.** If a package is installed outside the installer (operator runs apt manually), the inventory diverges. v0.3 doesn't reconcile; `stevens doctor` is a future place for "things on the system that the inventory doesn't know about" warnings.

  Outcome: 389/389 tests pass (1 OCR test skipped, unchanged). 92 new tests across migrations / approvals matcher / policy / dispatch / apt mechanism / system.* capabilities / installer agent / approvals CLI / end-to-end. Both e2e scenarios pass: BLOCKED → approve → replay → ok with audit `approval_via=per_call/<id>`; standing approval pre-loaded → silent execute with audit `approval_via=standing/<id>`.
  Deviations from plan:
  - **Steps merged into one commit.** Same shape as v0.1.6 and v0.2 — the milestone is the deliverable; shipping half doesn't compose cleanly.
  - **`stevens approval` / `stevens dep` CLI handlers built as standalone async functions (`cli_approvals.py`)** taking dependency-injected stores. The Postgres-backed `ApprovalStore` implementation isn't wired into the existing `cli.py` yet — that wiring is operator-side and needs a real DB connection. Tests exercise the handlers against in-memory stores, the load-bearing part. Production wiring is a v0.3.1 follow-up after a Postgres-backed `ApprovalStore` lands.
  - **`stevens dep remove` deferred to v0.3.x.** The rollback plan is recorded at install time; running it needs an operator-side SecurityClient (with a privileged operator key). Not blocking — for now, operators run `system.execute_privileged` directly with the recorded rollback plan.
  - **Standing-approval matcher requires the call to echo plan metadata.** The installer's `system.execute_privileged` call now also echoes `mechanism`, `packages`, `source` so standing-approval predicates can gate on them. The broker re-validates against the stored plan via `plan_id`, so these echoed values can't widen the privilege; they're matcher metadata only. Documented inline in `agents/src/agents/installer/agent.py`.
  - **`requires_approval` check before capability dispatch.** Approval-gating happens BEFORE the capability is invoked, so a misconfigured / missing handler doesn't slip through; BLOCKED is returned cleanly.
  - **No inventory row on remove/purge.** When the operation is remove or purge, the existing inventory row is `mark_removed`-ed rather than a new row appended. The inventory tracks "what's installed now," not "every action taken."
  - **Replay envelope carries `replay_request_id` field.** Folding the replay marker into the signed envelope (vs a side channel) keeps the request signed and auditable. The dispatcher's `bypass_approval_for_request_id` predicate gates on it.
  Surprises:
  - **`StandingApproval` is frozen — revoke produces a new instance.** Tests caught this; first attempt mutated `revoked_at` directly. Production Postgres-backed implementation will just `UPDATE`, no Python-level immutability concern.
  - **`_summarize_params` is the operator-facing "what is this asking for?" line.** Crucial UX surface — what Sol reads when deciding to approve. v0.3 includes mechanism / packages / account_id / thread_id; this surface grows as new capabilities use approval gates.
