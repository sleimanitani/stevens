# Build Plan — v0.3.2-postgres: production wiring for the v0.3 primitives

**Goal:** make `v0.3-installer-and-approvals` runnable in production. Postgres-backed implementations of `ApprovalStore`, `PlanStore`, `Inventory`. Enkidu loads them at boot; the CLI talks to Postgres directly. Migration runner ships as a script. Once this lands, `stevens dep ensure tesseract-ocr` is a real working command.

**Status:** in-progress — started 2026-04-30
**Charter refs:** STEVENS.md §3 (Security architecture); `docs/protocols/approvals.md`, `docs/protocols/privileged-execution.md`.
**Predecessors:** v0.3-installer-and-approvals (`c2f0929`), v0.3.1-web (`8d1f64a`).
**Blocks:** real `dep ensure` calls; real `stevens approval` flow; first real Email PM run via `stevens onboard gmail` (which inserts a `channel_accounts` row).

---

## Scope

In this milestone:

1. `security/.../approvals/store_postgres.py` — Postgres-backed `ApprovalStore` implementing the existing Protocol.
2. `security/.../system_runtime_postgres.py` — Postgres-backed `PlanStore` and `Inventory` for the installer agent's privileged-execution path.
3. Wire into Enkidu boot (`__main__.py`): construct Postgres-backed impls; load all active `standing_approvals` rows into the in-memory matcher.
4. Wire into the operator CLI (`cli.py`): `stevens approval *` + `stevens dep *` connect to Postgres directly (in-memory implementations remain test-only).
5. **Refresh signal**: `_admin.refresh_approvals` capability + a thin admin client. When the operator revokes a standing approval (or grants one), Enkidu reloads its matcher from the DB.
6. `scripts/db_migrate.sh` — applies all `resources/migrations/*.sql` in order against `$DATABASE_URL`. Idempotent (uses `IF NOT EXISTS` from the migration files).
7. Smoke tests for the Postgres implementations — declared manual; require `STEVENS_TEST_DATABASE_URL` pointing at a disposable DB.

Out of scope:
- Replacing the in-memory test implementations (those stay; they're the unit-test surface).
- `pytest-postgresql` integration. Future, when a fixture is needed.
- Postgres for the v0.3.1 web cache (still in-memory; documented as a future shape).
- Operator-deploy automation — bringing up `docker compose up postgres`, applying migrations, etc., is documented in DEVELOPMENT.md but not orchestrated by code.

### Step 5 — Refresh signal [x]

When the operator runs `stevens approval standing grant` or `revoke`, Enkidu's in-memory matcher needs to drop / add the entry. v0.3.2 ships the simplest version: an admin capability that triggers a reload.

**Files:**
- `security/src/stevens_security/capabilities/admin.py` — `_admin.refresh_approvals` capability. When invoked, Enkidu's dispatcher (or the capability handler with access to the matcher via context) reloads the matcher from the store. The `_admin.*` namespace is allowed only for an `operator` caller (not for any agent).
- `cli.py` — after `grant` / `revoke`, the CLI calls the admin capability via the SecurityClient if Enkidu is running. If Enkidu is not running, just write the DB; next boot picks up the change.
- An "operator" agent identity must exist in `agents.yaml` (Sol generates it once via `stevens agent provision operator`). The CLI uses the operator's keypair (env: `STEVENS_OPERATOR_PRIVATE_KEY_PATH`).

**Design note**: putting the refresh on a UDS capability call means the operator CLI has the same trust model as any other broker caller. Privilege escalation via the CLI is exactly what we'd guard against; using the same identity layer keeps it consistent.

## Risks

- **Connection pool sizing** under cold start. `shared.db.get_pool()` uses min=1, max=10 — should be fine for our scale, but Enkidu's boot loads all standing approvals which is one query, and dispatcher writes are at-most-one query per call. Re-validate if approval volume grows.
- **JSONB round-trip for predicates.** `predicates` is JSONB; psycopg3 handles `json.loads/dumps` automatically with the right type adapter. Verify predicate equality survives the round-trip with a test (`{"mechanism": "apt"}` written → read returns the same dict).
- **Schema drift**: `db_migrate.sh` runs every migration every time, but `IF NOT EXISTS` makes it idempotent. If a migration is destructive (it shouldn't be — we're append-only), this approach breaks. Documented in the migrations' headers.
- **Refresh signal race**: `revoke` writes the DB row → notifies Enkidu → Enkidu reloads. Between write and notify there's a window where Enkidu's cache is stale; an in-flight call could match the just-revoked approval. v0.3.2 accepts this race — it's bounded to the round-trip of the admin capability call (sub-millisecond local UDS). v1.0 mitigation if needed: have the matcher consult the DB on every match-success to confirm not-revoked.
