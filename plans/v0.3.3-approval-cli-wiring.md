# Build Plan — v0.3.3-approval-cli-wiring

**Goal:** close the v0.3.2 gap. The `_admin.refresh_approvals` and `_admin.mark_request_approved` capabilities exist; the CLI doesn't yet auto-invoke them after grants/revokes/approvals. Also: borrow OpenClaw's approval-forwarding config (session/targets/both) so approval requests can be routed to a designated chat channel rather than only the CLI.

**Status:** in-progress — started 2026-05-01
**Charter refs:** `docs/protocols/approvals.md`, `docs/architecture/agent-isolation.md`.
**Predecessors:** v0.3.2-postgres (`902f0d8`).

---

## Scope

1. **Operator identity provisioning helper** — small wrapper that ensures an `operator` agent identity exists with a 0o600 key file at `~/.config/demiurge/agents/operator.key`. Used by the CLI to sign admin-capability calls.
2. **Auto-invoke `_admin.refresh_approvals`** from `demiurge approval standing grant/revoke`. Best-effort: if Enkidu isn't running (UDS missing), skip with a warning.
3. **Auto-invoke `_admin.mark_request_approved`** from `demiurge approval approve`.
4. **OpenClaw-shaped forwarding config** — when a per-call approval lands, optionally publish a notification event (`approval.requested.<id>`) so a future channel-bound consumer can route it to a chat thread. Config lives in `security/policy/approval_forwarding.yaml` with the OpenClaw shape: `mode: session | targets | both`, `targets: [{channel, account_id, thread_id?}]`, `agent_filter: [...]`.
5. The actual chat-channel publication is left for a future "approval forwarder" agent — v0.3.3 just emits the event with the right shape.

Out of scope:
- A real chat-channel publisher (future agent).
- Approval timeout / auto-expire.
