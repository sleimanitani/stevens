# Build Plan — v0.4-sphinx (PDF agent)

**Goal:** ship Sphinx, the PDF agent. Inspects each PDF + the request, picks a strategy (pdfplumber / tesseract OCR / Docling for complex layout / etc.), executes, returns. Pattern lifted directly from Hermes's `terminal_tool.py` / `web_tools.py` factory model: normalize config → check prereqs → resolve state → instantiate. Both sync (skill for ReAct) and async (Sphinx for event-driven) paths converge on the same strategies.

**Status:** in-progress — started 2026-05-01
**Predecessors:** v0.3.4-arachne-upgrade (`d8c2b2b`).
**Charter refs:** STEVENS.md §1.1 (Names) for Sphinx; `docs/architecture/agent-isolation.md`.

---

## Architectural posture

Same dual-path shape we landed on for Arachne:

- **Synchronous path** — ReAct agents call the existing `read_pdf` skill. Today's skill is a single-strategy pdfplumber+OCR fallback. We upgrade it to dispatch via the new strategy router. No bus round-trip; for trivial cases the strategy decision is sub-millisecond.
- **Async path** — agents publish `pdf.parse.requested.*`; Sphinx subscribes, runs the strategy router under a worker pool, publishes `pdf.parse.response.*`. For heavy strategies (Docling loads ML models) async is the right shape because cold-start dominates.

Both paths share the **strategy registry**. Adding a new strategy is one new module under `skills/src/skills/tools/pdf/strategies/<name>.py`; both paths pick it up automatically.

---

## Scope

1. **`PDFStrategy` Protocol + dispatcher** at `skills/src/skills/tools/pdf/dispatcher.py`. Hermes-pattern: `inspect → normalize → check prereqs → resolve state → instantiate`. Output: a chosen strategy + a reason string for observability.
2. **Three strategies in v0.4:**
   - `native_text` — pdfplumber, today's default. Fast, text-layer PDFs.
   - `ocr_fallback` — pdfplumber + pytesseract (today's fallback path), for scanned PDFs.
   - `docling` — IBM Docling, ML-based, for complex layouts/tables/formulas. Lazy-imported (the model load is heavy); auto-skipped if Docling isn't installed.
3. **`read_pdf` skill rewrite** — same input shape, new dispatch internals. Adds optional `prefer_strategy: <name>` and `request_hint: <str>` arguments so callers can bias the router.
4. **Sphinx agent** — `agents/src/agents/pdf/agent.py` (display name **Sphinx**, code id `pdf`). Subscribes to `pdf.parse.requested.*`. 4-worker semaphore (matches Arachne).
5. **Event schemas** — `PDFParseRequestedEvent`, `PDFParseResponseEvent` in `shared.events`.
6. **STEVENS.md §1.1** — add Sphinx to the names table.

Out of scope:
- LLM-based strategy selection. v0.4 uses deterministic rules (page-count, has-text-layer, file-size, presence of tables). Smarter routing is a v0.5 concern.
- Per-page strategy mixing (e.g., text-layer for pages 1–10, OCR for pages 11+). Strategies operate on the whole PDF in v0.4.
- pymupdf, unstructured.io as alternate strategies — Protocol supports them; they land when needed.

### Step 1 — Strategy Protocol + dispatcher [x]

**Files:**
- `skills/src/skills/tools/pdf/strategies/__init__.py` — `PDFStrategy` Protocol (`name`, `available()`, `inspect()`, `extract(path, mode, options)` → result dict matching the existing `read_pdf` shape), registry (`register`, `get`, `known`), `select_strategy(inspection, hint=None, prefer=None) -> PDFStrategy` (the router).
- `skills/src/skills/tools/pdf/dispatcher.py` — `dispatch(path, *, mode, hint=None, prefer=None) -> dict`. Inspects the PDF cheaply (page count, has_text_layer, has_images, file_size, encrypted? — uses pdfplumber's metadata only, no full extract), picks a strategy, runs it, attaches `strategy_used` + `decision_reason` to the result.
- `skills/tests/test_pdf_dispatcher.py` — 6 tests: text-layer PDF picks native_text; scanned-empty-text PDF picks ocr_fallback; tables-spanning-pages with hint="extract tables" prefers docling if available; explicit prefer wins; encrypted PDF returns the structured error before strategy selection; PDF over MAX_PAGES warns.
