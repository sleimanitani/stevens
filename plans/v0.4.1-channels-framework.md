# Build Plan — v0.4.1-channels-framework

**Goal:** extract OpenClaw's channel-adapter abstraction into a Demiurge-shaped framework so future channel adapters (Signal / iMessage / Slack / Discord / Telegram / Matrix) cost ~half what they otherwise would. v0.4.1 ships the framework and **does not port a new channel** — that's a v0.4.x follow-up gated on which channel Sol picks first.

**Status:** in-progress — started 2026-05-01
**Predecessors:** v0.4-sphinx (`b9ef5b1`).
**OpenClaw reference paths** (from `~/oss-references/openclaw`): `src/channels/session.ts`, `extensions/{slack,signal,discord}/src/channel.ts`, `src/cron/isolated-agent/run.ts` (the agent-side pickup of channel routes).

---

## What we're borrowing from OpenClaw

The core seams worth porting (per the read-through report):

1. **`ChannelRoute`** — normalized `(channel_type, account_id, target_id, thread_id?)` tuple. The single shape every adapter produces inbound and consumes outbound, regardless of provider quirks.
2. **`ChannelSession`** — per-thread state envelope: route + recent message metadata + pending action queue + delivery hints (typing indicator capable, edit capable, max chunk size, supports modals).
3. **`OutboundAdapter`** — every adapter implements `send(session, content_kind, payload) -> delivery_id`. Content kinds: `plain_text`, `markdown`, `chunked` (long messages broken up), `block` (rich UI when supported), `approval_prompt`, `typing` (start/stop). Adapter declares which kinds it supports; framework synthesizes the closest fallback for unsupported ones.
4. **Streaming primitives** — `start_message`, `append_chunk`, `finalize` mapped to provider capability (Discord/Slack: edit-in-place; Signal: re-send with progress markers).

We **do not** borrow:
- OpenClaw's per-channel auth-flow specifics (we already have Enkidu-mediated credential storage; their per-channel filesystem creds don't fit).
- Their channel-as-persona binding model (we have subject agents that cut across channels — covered in `CLAUDE_openclaw_evaluation.md`).
- Their per-call approval-message-in-chat shape (we have the v0.3.3 approval-forwarding event for that, generalized beyond chat).

---

## Scope

1. **`shared/channels/` package** — the framework. New top-level `shared/src/shared/channels/` since this is plumbing every channel uses.
2. **`ChannelRoute` + `ChannelSession`** dataclasses with the normalized shape.
3. **`OutboundAdapter` Protocol** + `ContentKind` enum (`plain_text`, `markdown`, `chunked`, `block`, `approval_prompt`, `typing_start`, `typing_stop`).
4. **`AdapterCapabilities`** dataclass — declared per-adapter (which content kinds, max chunk size, edit-in-place support).
5. **Synthesis layer** — given a content kind the adapter doesn't support, pick the closest one it does (e.g., `block` → fall back to `markdown` → fall back to `plain_text`; `chunked` → split into multiple `plain_text` messages).
6. **Streaming protocol** — `StreamingDelivery` class wrapping start/append/finalize, mapped to provider capability via the adapter.
7. **Reference adapter** — a tiny `EchoAdapter` that just records what it would have sent. Used by tests.
8. **Migrate ONE existing adapter onto the framework** — Gmail. The Gmail adapter today doesn't really do "send" (drafts only) but it does structure inbound events; we'll show how its inbound side maps to `ChannelRoute` so the pattern is concrete.

Out of scope:
- Porting Calendar / WhatsApp Cloud onto the framework (existing adapters keep working unchanged; we migrate them when we touch them next).
- Porting a new channel (Signal etc.) — separate v0.4.x milestone.

## Steps

> `[ ]` todo · `[~]` in progress · `[x]` done.

### Step 1 — Core shapes [x]

**Files:**
- `shared/src/shared/channels/__init__.py` — re-exports for the public surface.
- `shared/src/shared/channels/route.py` — `ChannelRoute` (frozen dataclass: channel_type, account_id, target_id, thread_id?, peer_id? for DMs).
- `shared/src/shared/channels/session.py` — `ChannelSession` (frozen dataclass: route + recent_messages: List + capabilities: AdapterCapabilities + extra: dict).
- `shared/tests/test_channels_route.py` — 4 tests: equality, hashing in a set, round-trip-through-dict, channel-type validation.

### Step 2 — OutboundAdapter Protocol + ContentKind [x]

**Files:**
- `shared/src/shared/channels/content.py` — `ContentKind` enum, `Content` payload union (typed dataclasses: PlainText, Markdown, Chunked, Block, ApprovalPrompt, TypingStart, TypingStop).
- `shared/src/shared/channels/adapter.py` — `AdapterCapabilities` (frozenset of supported kinds + max_chunk_chars + supports_edits + supports_threads), `OutboundAdapter` Protocol (`send(session, content) -> DeliveryRef`), `DeliveryRef` (channel-side message/thread id + provider hint).
- `shared/src/shared/channels/synthesis.py` — `synthesize(content, capabilities) -> Content` — picks closest supported kind. Fallback chain: block → markdown → plain_text; chunked of N chars → split into N // max_chunk_chars plain_text. Approval prompt with rich UI capability → block; without → markdown with a structured "reply with /approve <id>" line.
- `shared/tests/test_channels_adapter.py` — 8 tests: capabilities check, synthesis fallbacks (block→markdown, markdown→plain_text, chunked→split), approval_prompt synthesis with/without block support.

### Step 3 — Streaming primitives [x]

**Files:**
- `shared/src/shared/channels/streaming.py` — `StreamingDelivery` (start, append_chunk, finalize). For adapters with `supports_edits`, accumulates and edits a single message in place. For adapters without, sends each chunk as a separate message (with a header marker).
- `shared/tests/test_channels_streaming.py` — 4 tests: edit-in-place adapter coalesces, non-edit adapter sends multiple, finalize closes the stream, mid-stream error surfaces.

### Step 4 — EchoAdapter reference [x]

**Files:**
- `shared/src/shared/channels/echo.py` — `EchoAdapter` records every send into an in-memory log. AdapterCapabilities supports all kinds (so synthesis is a no-op when testing it). Used by tests + future smoke checks.

### Step 5 — Gmail inbound migration [x]

**Files:**
- `channels/gmail/src/gmail_adapter/route.py` — `route_for_inbound(email_event) -> ChannelRoute`. Pure helper; doesn't change Gmail's actual event-publishing path. The point: show the framework integrating with at least one real adapter so the pattern's concrete and we can migrate the rest opportunistically.
- `channels/gmail/tests/test_route.py` — 2 tests: route from a personal account, route preserves thread_id.
