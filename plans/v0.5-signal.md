# Build Plan — v0.5-signal

**Goal:** Signal channel adapter on the v0.4.1 framework. First new channel that proves the framework. Talks to a `signal-cli-rest-api` daemon (the standard wrapper around signal-cli) over HTTP — the daemon owns the linked Signal session; we never touch Signal's encryption directly.

**Status:** in-progress — started 2026-05-01
**Predecessors:** v0.4.1-channels-framework (`31f464f`).
**Reference:** OpenClaw's `extensions/signal/` (~50 files of TS) for the integration patterns; we're not copying code, just borrowing shape.

---

## Operator dependency: signal-cli-rest-api daemon

Signal's protocol is non-trivial; the standard solution is to run [`signal-cli-rest-api`](https://github.com/bbernhard/signal-cli-rest-api) — a small REST wrapper around `signal-cli`, packaged as a Docker container. It owns the linked Signal session and exposes:

- `GET  /v1/qrcodelink/{device_name}?number={phone}` — get a QR code to link a new device
- `POST /v2/send` — send a message
- `GET  /v1/receive/{number}` — receive new messages (polling)
- `GET  /v1/about` — health check

For v0.5 we **assume the daemon is running and reachable** (operator brings it up via compose). v0.5.x can add a one-shot installer via the v0.3 installer agent (`demiurge dep ensure signal-cli-rest-api` → installs the Docker image + sets up service).

## Scope

1. Migration `009_signal_channel_type.sql` — adds `signal` to the `channel_accounts.channel_type` allow-list.
2. `channels/signal/` package — Python adapter on the v0.4.1 framework.
3. `signal_adapter/client.py` — thin HTTP client around signal-cli-rest-api.
4. `signal_adapter/adapter.py` — `SignalAdapter` implementing the `OutboundAdapter` Protocol from `shared/channels/`.
5. `signal_adapter/route.py` — `route_for_inbound(received_msg) -> ChannelRoute` (mirror of the Gmail helper from v0.4.1).
6. `signal_adapter/main.py` — FastAPI app for the inbound poller + management endpoints (matches the existing channels' shape).
7. `signal_adapter/add_account.py` — `demiurge onboard signal` wired path: phone + daemon URL → QR link printed → polls until linked → stores phone in sealed store + inserts `channel_accounts` row.
8. `shared.events.SignalMessageEvent` — event schema for inbound messages.
9. `compose.yaml` — add `signal-cli-rest-api` service + the new `signal-adapter` service.
10. `demiurge onboard signal` — wire into the existing onboarding CLI.
11. DEMIURGE.md §1.1 — Signal stays unnamed for now (it's an adapter, not a reasoning agent).

Out of scope:
- Group message handling beyond the basic shape (groups work; advanced group admin actions are a follow-up).
- Media (images / files) — text only in v0.5.
- Disappearing-message timer setting from our side.
- Real signal-cli-rest-api integration tests (declared manual; daemon needs to actually be running with a linked phone).
