# Build Plan — v0.6-google-wizard

**Goal:** shrink Google (Gmail + Calendar) operator-onboarding from ~8 manual steps to ~3 by driving every GCP step that has a REST API via `gcloud` CLI, and walking the operator through only the irreducible manual ones.

**Status:** in-progress — started 2026-05-01
**Predecessors:** v0.5-signal (`0251c06`).
**Charter refs:** OpenClaw audit memory (we deliberately don't borrow OpenClaw's pattern — Google won't bless us as it does Gemini CLI). Closest peer: the existing `demiurge onboard gmail` runbook.

---

## What's automatable vs manual

GCP onboarding decomposes:

| Step | API? | Done by |
|---|---|---|
| Create project | yes (Resource Manager) | wizard |
| Enable APIs (Gmail, Calendar, Pub/Sub) | yes (Service Usage) | wizard |
| OAuth consent screen → External + Production + scopes | **no** | operator (wizard guides) |
| Create Desktop OAuth client | **no** | operator (wizard guides) |
| Download client_secret.json | **no** | operator (wizard waits for file) |
| Create Pub/Sub topic | yes | wizard |
| Grant `gmail-api-push@…` Publisher | yes (IAM) | wizard |
| Create Pub/Sub push subscription | yes | wizard (operator supplies webhook URL) |
| Operator does `demiurge onboard gmail` | n/a | chains automatically |

The three irreducible manual steps are deliberate — Google doesn't expose APIs for OAuth consent screen / Desktop client / JSON download because they're trust-establishing actions. We can't shortcut them, but we can make them painless by opening the right Cloud Console URL with the right query params at the right moment and printing exact field values.

---

## Scope

1. **`demiurge wizard google`** — single new top-level CLI subcommand.
2. **`gcloud` subprocess driver** in `security/src/demiurge/wizards/google.py`. Detects gcloud presence; clear "install gcloud first" message if missing. Detects `gcloud auth login` status; runs the operator through it if not.
3. **Project + API enable + Pub/Sub topic + IAM grant + push subscription** all driven via `gcloud` subprocess. Wizard shows the exact command + asks `[y/N]` before each side-effecting call.
4. **Manual-step guidance** — for OAuth consent screen, OAuth client creation, and JSON download, the wizard:
   - Prints the exact Cloud Console URL to open (deep-links into the right page).
   - Prints exact field values to enter (app name, scopes, redirect URI, etc.).
   - Polls for the downloaded `client_secret*.json` in `~/Downloads/` (or operator-specified path).
5. **Webhook URL collection** — wizard asks operator for their public push-receiver URL (Tailscale Funnel etc.). Stores it in the channel_accounts metadata for the operator to update later if it changes.
6. **Chain into `demiurge onboard gmail`** — once the JSON is detected, ingests it via the existing onboarding flow without an extra command.
7. **Idempotent re-runs** — wizard detects existing project / topic / etc. and offers to reuse rather than re-create. Operator can re-run safely.

Out of scope:
- Calendar-specific wizardry beyond what Gmail covers (Calendar uses the same OAuth client + project; no separate setup other than the per-account flow which `demiurge onboard calendar` already handles).
- Auto-creating the OAuth consent screen via undocumented APIs (some open-source projects scrape this; we won't — it's brittle and against ToS).
- Service-account flows (we use OAuth user delegation).
- Workspace-only "Internal" consent screens — Sol uses both personal + Workspace; External + Production is the right pick (locked in earlier).

## Risks

- **gcloud version drift.** Some commands change flags between versions. Pin the wizard's `gcloud --version` minimum and surface a clear "upgrade gcloud" message if the operator's is too old.
- **OAuth consent screen verification.** With sensitive scopes (`gmail.modify` etc.), Google may require verification for "External + Production" if the user count exceeds 100. For Sol's single-user case this won't bite. Documented in the wizard output.
- **Push subscription URL must be publicly reachable.** If the operator's webhook URL isn't reachable from Google, the subscription creates fine but receives nothing. Wizard does a one-shot reachability check (HEAD request from our process to the URL) and warns if it fails — though that's still no guarantee Google can reach it.
- **Idempotence races.** If the operator re-runs the wizard while a prior run is still creating the topic, the second create call 409s. Wizard treats 409 as "already exists, reuse" rather than fail.
