# Build Plan — v0.7-janus

**Goal:** ship Janus — operator-assisted browser automation that drives the OAuth/config dance for any provider with a web dashboard. Operator owns the browser session (signs in, approves OAuth, solves CAPTCHAs); Janus does everything else (nav, fill, click, extract token, store in sealed store).

**Status:** in-progress — started 2026-05-01
**Predecessors:** v0.6-google-wizard (`c6cd4da`).
**Reference shape:** Hermes's `agent-browser` + Playwright pattern (per the read-through report); strategy-router pattern from Sphinx (`v0.4`).

---

## Architectural posture

**Operator-assisted, not autonomous.** Janus never stores or types the operator's account credentials. The operator signs in once at the start of each recipe, then watches Janus click through the form-fill steps. CAPTCHAs / 2FA / phone verification → Janus pauses and surfaces "do this in the browser, then press Enter here." Anti-automation evasion is explicitly out of scope.

**Strategy-router pattern.** Each provider is a `PlatformRecipe` (Protocol) — same shape as Sphinx's PDF strategies and the search backends. `available()` checks prereqs (Playwright installed); `steps()` returns the ordered list of `Step` objects; `execute(session, ...)` runs them and returns the extracted secret. Adding a new provider is one new recipe module.

**CLI shape, not agent (yet).** v0.7 ships `stevens janus <recipe>` as a CLI subcommand. The operator runs it from their terminal on the machine with a real display + sign-in cookies. v0.8 can wrap it in a bus-subscribed agent ("OperatorNeedsCredsForX → trigger Janus recipe X") when there's a consumer that wants the async path.

## Scope

1. **`security/.../wizards/janus/`** package.
2. **`Step` types** (dataclass union): `Nav`, `WaitFor`, `Fill`, `Click`, `OperatorAction` (pauses for human), `Extract` (pulls text from a selector into a named slot), `StoreSecret` (writes a slot to the sealed store).
3. **`PlatformRecipe` Protocol** — `name`, `description`, `available()`, `prerequisites` (e.g. "have an Anthropic account"), `steps()`, `execute(session, slots) -> ExecutionResult`.
4. **`BrowserSession` Protocol + Playwright impl** — `nav`, `wait`, `fill`, `click`, `extract`. Tests use a `FakeBrowserSession` that records calls + returns canned values.
5. **Three recipes for v0.7:**
   - `brave` — sign up / log in to brave.com/search/api → create key → copy → store as `web.brave.api_key`.
   - `anthropic` — log in to console.anthropic.com → API keys → create → copy → store as `compress.anthropic.api_key`.
   - `google_oauth_client` — drives the manual half of v0.6: OAuth consent screen → External + Production + scopes → Create credentials → Desktop client → download. Doesn't bypass the actual download (Google forces a real browser save dialog), but eliminates every click before it.
6. **`stevens janus <recipe>`** CLI subcommand. Auto-detects which sealed-store secret would be set; refuses to overwrite without `--rotate`.
7. **Doctor integration** — `stevens doctor` adds a check: "Playwright browsers installed?" with one-line remediation (`uv run playwright install chromium`).
8. **Recipe registry** in the same shape as the search-backend / PDF-strategy registries — recipes register at module import.

Out of scope for v0.7:
- Slack / Discord / Telegram / Meta recipes (queued for v0.7.x follow-ups; their auth shapes are each their own thing).
- Headless mode. Janus is operator-watched by design.
- Remote browser providers (Browserbase, Browser Use). Local Playwright only in v0.7; remote is v0.8.x.
- The bus-subscribed Janus-agent shape. CLI only.

## Risks

- **Provider UI churn.** Brave / Anthropic / Google update their dashboards; recipes break. Mitigation: every recipe has selectors in one place at the top of its module; failed selector = clear "recipe out of date for $provider, see Janus README" error rather than silent click of the wrong button.
- **Playwright installs are big.** ~150MB Chromium download on first `playwright install chromium`. Documented; doctor check surfaces it.
- **Anti-automation detection.** Google in particular flags new sessions sometimes. Mitigation: use a persistent context dir (`~/.config/stevens/janus-profile/`) so the operator's "this device is OK" bit sticks across runs.
- **Operator's account credentials.** Janus NEVER stores them. The operator signs in via the visible browser; Playwright's persistent context preserves the session cookie (encrypted-at-rest by the OS keyring on macOS; on Linux, just file perms). If the operator wants stronger isolation, they delete the profile dir between recipes.
- **The Google recipe still needs the operator at the consent step.** Google forces several "Save and Continue" buttons that include CAPTCHA-like checks. Janus clicks the obvious ones; surfaces the rest.

## Acceptance criteria

- [ ] `stevens janus brave` walks the operator from a brand-new Brave account to a stored `web.brave.api_key` in the sealed store, with operator only doing: sign in, click "I accept", solve the one CAPTCHA if it appears.
- [ ] `stevens janus anthropic` does the same for `compress.anthropic.api_key`.
- [ ] `stevens janus google_oauth_client --project-id <id>` chains in after `stevens wizard google` and eliminates the OAuth-consent + client-create manual steps.
- [ ] All recipes resume correctly when re-run (idempotent — detects existing keys and refuses without `--rotate`).
- [ ] `stevens doctor` surfaces "playwright browsers not installed" with the install command.
- [ ] Existing 573 tests still pass; ≥20 new tests covering Step / Recipe / Runner / Doctor logic with FakeBrowserSession.

**Declared manual:**
- Real recipe runs against real provider sites — operator runs them; no automated test against live UIs (would be flaky and abuse the providers).
- Playwright browser install (`uv run playwright install chromium`) — operator runs once.
